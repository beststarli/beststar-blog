# 港口水环境面试问题挖掘
## 字节跳动前端校招项目面试问题库

> 基于 `package/client` 与 `package/server` 的实际代码整理。个人主线是案例管理、文件管理、计算任务、SSE、虚拟列表及 Node.js 中台。WebGL、GLSL、粒子流场和自定义渲染图层不属于个人开发范围。

---

## 一、面试前先统一项目边界

### 推荐的个人职责口径

> 我主要负责 FVCOM 案例、模型文件和计算任务这条前后端链路。前端使用 React、TypeScript 和 Zustand 管理跨组件业务状态，通过 SSE 展示长耗时任务进度，并针对文件列表做虚拟化；Node.js 中台使用 Fastify、Prisma 和 RustFS，完成案例持久化、大文件流式传输及计算服务编排。WebGL 流场渲染由其他成员负责，我负责向渲染模块提供结果对象列表、资源地址和区域范围。

### 被问到 WebGL 时

> WebGL 和粒子流场不是我实现的，所以我不会把渲染算法作为个人亮点。我与该模块的接口边界是：计算任务完成后，由中台提供结果文件列表、对象 URL 和区域 bounds，渲染模块消费这些数据。我可以重点展开状态同步、大文件链路或前端性能优化。

不要回答“我不懂 WebGL”。更准确的表达是“没有负责内部实现，但理解上下游接口和模块边界”。

## 二、第一优先级：几乎一定会问

### 1. 请用一分钟介绍这个项目，以及你具体负责什么

**考察点**：表达结构、个人贡献、是否把团队成果算到自己头上。

**回答抓手**：

1. 业务问题：科研人员依赖本地文件和命令行运行模型。
2. 平台链路：案例创建、区域选择、文件上传、计算执行、进度追踪、结果访问。
3. 个人模块：Zustand 状态、SSE 任务面板、虚拟列表、Fastify/RustFS 中台。
4. 最难问题：不同生命周期状态协同、大文件传输、长任务实时状态。
5. 明确 WebGL 非个人职责。

### 2. 这个项目最难的技术问题是什么？

建议回答“长耗时任务的状态一致性”，而不是“页面很多”。

可展开：

- 浏览器局部状态、数据库任务状态、计算服务实际状态是三个状态源。
- 刷新页面、切换案例、取消任务、任务完成都可能造成状态漂移。
- 采用数据库作为事实来源，SSE 首次发送快照，之后发送状态变更。
- 前端将当前案例与关注任务分离，避免切换案例导致任务丢失。
- 当前仍缺任务版本号和严格的取消确认，是后续改进点。

### 3. 为什么选择 Zustand，而不是 Redux 或 Context？

**回答抓手**：

- 状态跨越地图、设置面板、案例列表、任务面板，但状态模型不需要 Redux 的完整样板和中间件体系。
- Zustand 支持组件通过 selector 订阅字段，不要求额外 Provider。
- Context 更适合低频、相对稳定的全局依赖；单个大 Context 更新容易扩大重渲染范围。
- 上传 loading、滚动位置等只属于组件自身，仍保留为局部状态，没有把所有状态都塞进 Zustand。

**可能追问**：

- selector 为什么能减少渲染？
- 一次选择多个字段时如何避免新对象导致重渲染？
- Zustand 状态在刷新后为什么会消失？
- 哪些状态应该由服务端状态库管理？

### 4. 你是怎么划分全局状态和局部状态的？

**回答抓手**：

- 跨页面/跨面板共享、需要多个组件修改的领域状态进入 Store。
- `selectedCaseID`、`watchedTaskIds` 属于业务状态。
- `uploading`、`scrollTop`、列表容器高度属于单组件交互状态。
- 服务端可恢复的任务状态不应只保存在 Store，数据库才是事实来源。
- 一次性事件用 trigger 计数器表达，但项目扩大后可收敛为 action 或缓存失效机制。

### 5. 为什么用 SSE，不用 WebSocket 或轮询？

**回答抓手**：

- 任务进度主要是服务端到浏览器的单向文本推送。
- SSE 基于 HTTP，浏览器原生提供 `EventSource`，实现成本较低。
- 相比固定轮询减少无状态变化时的重复请求。
- 相比 WebSocket 更适合低频、单向事件。
- 若需要高频双向控制、二进制数据或协同编辑，再考虑 WebSocket。

**必须知道的边界**：

- `EventSource` 默认只直接支持 GET。
- 原生 `EventSource` 不方便添加任意请求头。
- SSE 需要正确设置 `text/event-stream`、禁用缓存并保持连接。
- 当前统一任务接口事件驱动，但单任务接口仍存在服务端定时查询数据库，不能声称完全无轮询。

### 6. SSE 连接断开、页面刷新后如何恢复？

**当前实现**：

- 连接建立时先查询数据库并发送当前任务快照。
- 后续监听 `CASE_PROGRESS` 事件，状态变化时再次推送。
- 前端刷新后根据 `running` 状态恢复关注任务。
- 前端卸载时执行 `EventSource.close()`。
- 服务端监听连接关闭并移除 `EventEmitter` listener。

**改进方向**：

- 增加心跳，避免代理因长时间无数据关闭连接。
- 使用事件 ID 和 `Last-Event-ID` 补发缺失事件。
- 增加指数退避、连接状态 UI 和错误监控。
- 多实例部署使用 Redis Pub/Sub 或消息队列，而不是进程内 `EventEmitter`。

### 7. 为什么使用虚拟列表？你是怎么实现的？

**回答抓手**：

- 完整列表的 DOM 数量随文件数量 O(N) 增长。
- 当前使用固定行高 `36px`。
- 根据 `scrollTop` 和容器高度计算首尾索引。
- 首尾各增加 `overscan=6`，降低快速滚动时短暂白屏。
- 外层保留 `数据量 × 行高` 的占位高度，实际行绝对定位。
- `ResizeObserver` 监听容器高度变化，重新计算可见区。

**复杂度表达**：

实际渲染节点约为：

`可视区高度 / 36 + 2 × 6`

因此节点数主要与视口大小相关，而不是与总文件数相关。

### 8. 虚拟列表有哪些缺陷？如果行高不固定怎么办？

**回答抓手**：

- 当前方案依赖固定行高，动态内容会导致偏移计算不准确。
- 不定高列表需要测量实际高度并建立高度/偏移缓存。
- 某行高度变化后要更新后续累计偏移。
- 可使用二分查找定位首个可见项。
- 还需处理滚动锚定、快速滚动、键盘可访问性和焦点移出可视区。

### 9. 你做了懒加载，具体发生了什么？

**当前实现**：

- `/model` 和 `/fvcom` 使用 `React.lazy(() => import(...))`。
- `Suspense` 在异步 chunk 下载期间展示 fallback。
- Vite/Rollup 将动态 import 识别为拆包点。

**必须主动说明的真实结果**：

- 当前生产构建生成了 FVCOM 页面约 `66.08 kB gzip`、普通模型页约 `82.07 kB gzip` 的异步 chunk。
- 但入口仍约 `1,338.17 kB gzip`，说明公共依赖和 Store 引用让部分重依赖提前进入入口。
- 所以当前只能证明实现了路由拆包，不能声称首屏性能已经得到显著提升。

### 10. 为什么用了懒加载，入口包仍然很大？

这是很可能出现的字节式追问。

**回答抓手**：

- 路由组件异步不代表所有依赖都异步。
- `Router` 首屏直接引用 `layerStore`、`mapStore`，相关依赖可能把 Mapbox 链路带进入口。
- 首页使用 Ant Design Button，并一次性 import 多张模型图片。
- 公共依赖可能被 Rollup 提取到入口或共享 chunk。
- 全局引入 Mapbox CSS 也会增加首屏资源。

**优化方向**：

- 将地图相关初始化和 Store 清理下沉到模型路由内部。
- 避免入口模块静态引用重型业务模块。
- 对 ECharts、Mapbox 等重依赖进一步按功能动态加载。
- 首页非首屏图片使用 `loading="lazy"`，并转为 WebP/AVIF。
- 使用 bundle visualizer 分析模块组成，再决定 `manualChunks`，而不是盲目拆包。

### 11. Lighthouse 分数是怎么测的？为什么可信？

**正确回答方式**：

- 使用生产构建和 preview 服务，不测 Vite dev。
- 固定 Chrome 版本、页面、网络/CPU 模式和测试设备。
- 使用无痕窗口或清除存储。
- 优化前后各运行至少三次，记录中位数。
- 同时保留 HTML 报告。
- 不只看总分，还看 LCP、TBT、CLS 和资源体积。

**可能追问**：

- 为什么实验室数据和真实用户数据不同？
- Lighthouse Performance 为什么每次会波动？
- LCP、CLS、INP 分别是什么？
- 懒加载主要影响哪些指标？
- 虚拟列表为什么未必明显提升首屏 Lighthouse？

### 12. 为什么选择 Fastify？

**回答抓手**：

- 需求不仅是 CRUD，还包含类型化接口、multipart、Stream 和 SSE。
- Fastify 支持 Schema 驱动，配合 TypeBox完成运行时校验和静态类型推导。
- 插件与路由 prefix 适合按 `project`、`model`、`fvcom` 拆分模块。
- 对原生 response 和 Node Stream 操作直接，适合长连接与文件流。
- 性能是加分项，但不是唯一理由。

### 13. 为什么不用 Express？

> Express 完全可以实现，但接口 Schema、类型推导、运行时校验通常需要额外组合工具。Fastify 与 TypeBox 的组合更契合项目的类型化接口和模块化路由需求。选择不是因为 Express 做不到，而是为了减少接口约束和插件组织上的胶水代码。

**真实性边界**：

当前早期模块的 TypeBox 使用更完整，新增 FVCOM 部分仍有手写类型和 `any`，不要声称已经实现全链路端到端类型安全。

### 14. 大文件为什么要使用 Stream？

**回答抓手**：

- 如果先读取为完整 Buffer，内存占用随文件大小线性增长。
- Fastify multipart 提供文件可读流。
- S3 SDK `Upload` 可以消费流并写入 RustFS。
- 下载时将 RustFS `Readable` pipe 到 HTTP response。
- Stream 通过背压协调上下游速度，避免生产速度长期超过消费速度。

**基础追问**：

- Node.js Stream 有哪几类？
- `pipe` 和 `pipeline` 有什么区别？
- 什么是背压？
- `highWaterMark` 是什么？
- 流传输中途失败如何清理？

### 15. RustFS、PostgreSQL 和计算服务分别存什么？

**回答抓手**：

- PostgreSQL：案例 ID、名称、区域、文件对象键、任务状态、进度等结构化元数据。
- RustFS：输入文件和计算结果等非结构化大对象。
- 计算服务：真正执行 FVCOM 运算。
- Web 中台：校验案例、组织对象路径、调用计算服务、同步任务状态、向前端提供统一接口。

### 16. 为什么不把文件存在 PostgreSQL 或服务器本地磁盘？

**回答抓手**：

- 大对象进入数据库会增加数据库存储、备份和查询压力。
- 本地磁盘使服务实例与数据绑定，不利于扩容和迁移。
- 对象存储适合大文件、前缀组织和流式访问。
- 数据库保存对象键和业务关系，职责更清晰。

### 17. 对象存储和数据库如何保证一致性？

**当前边界**：

同名替换大致执行“识别旧对象 → 删除旧对象 → 上传新对象 → 更新数据库路径”，不具备跨系统原子事务。

**改进方案**：

- 先上传临时 key。
- 上传成功后，在数据库事务中切换元数据。
- 最后异步删除旧对象。
- 增加幂等键、失败补偿和定期对账。
- 记录对象状态，如 `uploading/valid/deleting`。

不要回答“使用 Prisma 事务就能保证 RustFS 和数据库一致”，Prisma 事务无法覆盖外部对象存储。

### 18. 任务状态机是怎么设计的？

**当前状态**：

`idle → running → completed/error`

取消后回到 `idle`。

**可能追问**：

- 两次点击执行如何避免重复任务？
- 旧任务事件会不会覆盖新任务？
- 取消成功是前端按钮复位，还是计算进程真的终止？
- 服务重启后如何恢复 `running` 任务？
- 为什么不增加 `queued/cancelling/cancelled`？

**改进方案**：

- 增加任务 ID/版本号，不只使用 caseID。
- 状态转换由服务端校验，使用条件更新防止非法转换。
- 启动接口使用幂等键。
- 取消需要计算服务确认，再写入最终状态。

### 19. 网络请求为什么设置超时和重试？

**当前实现**：

`extendFetch` 使用 `AbortController` 超时，并进行有限次数重试。

**必须主动指出风险**：

- 当前重试没有严格区分请求方法。
- POST 创建、上传、执行任务可能不是幂等的。
- 网络超时不代表服务端没有成功执行。

**改进方向**：

- 默认只重试 GET、HEAD 等幂等请求。
- 写操作通过业务幂等键去重。
- 按错误类型决定是否重试。
- 使用指数退避与随机抖动。
- 上传接口单独设计重试/续传机制。

### 20. 你如何证明这些优化真的有效？

**回答抓手**：

- 懒加载：对比 Vite 构建产物和首屏请求 waterfall。
- Lighthouse：对比 Performance、LCP、TBT、CLS，至少三次取中位数。
- 虚拟列表：构造固定数量文件，对比实际 DOM 节点数、Commit 耗时和滚动帧率。
- Zustand：用 React Profiler 查看组件渲染次数。
- Stream：在相同文件大小下记录服务进程峰值内存和传输耗时。

没有真实报告前，不要说具体提升百分比。

## 九、架构设计题

### 1. 如果同时有一千个计算任务，当前方案会遇到什么问题？

可从以下方面回答：

- EventEmitter listener 和 SSE 连接规模。
- 每次事件重新查询并发送全量活跃任务。
- 数据库频繁更新进度。
- 单进程内存状态无法跨实例。
- 计算服务并发限制和排队需求。
- 前端任务列表的增量更新与虚拟化。

改进为：

`任务 API → 队列 → Worker/计算服务 → 状态存储 → 消息通道 → SSE Gateway`

### 2. 如果服务需要多实例部署，怎么改？

- 任务状态继续落数据库或专用状态存储。
- EventEmitter 替换为 Redis Pub/Sub、Stream 或消息队列。
- SSE Gateway 订阅公共事件通道。
- 负载均衡层关闭响应缓冲并设置合理超时。
- 若连接状态只在实例内维护，需要考虑粘性会话或无状态重连。

### 3. 如果要求支持断点续传，怎么设计？

- 前端将文件切片。
- 服务端创建 uploadId。
- 并发上传分片，记录分片序号和 hash。
- 服务端或对象存储完成合并。
- 前端查询已上传分片后恢复。
- 最终校验整个文件 hash。
- 写操作需要幂等。

### 4. 如果要求任务进度绝对不能丢，SSE 足够吗？

SSE 只是传输通道，不负责可靠持久化。应将事件写入可靠日志或消息队列，分配单调事件 ID；客户端携带 `Last-Event-ID` 重连，服务端从对应位置补发。数据库最终状态仍用于兜底校准。

### 5. 如果把这个项目重新做一遍，你会改什么？

推荐回答：

1. 用明确的任务实体替代仅依赖 caseID 的状态。
2. 用服务端状态管理库收口请求缓存和失效，减少 trigger 字段。
3. 完善 TypeBox 覆盖，生成或共享前后端接口类型。
4. 引入可靠消息通道支持多实例任务进度。
5. 对上传做幂等、分片和一致性补偿。
6. 先建立 Lighthouse、bundle size、DOM 节点和内存基线，再持续优化。

## 十四、逐题口述参考答案

> 本章补齐前文中所有小问题的参考答案。回答时不必逐字背诵，建议先说结论，再结合本项目代码说明，最后补充边界或改进方案。

### A. 第一优先级问题补充答案

#### A1. 请用一分钟介绍项目

> 这是一个面向港口水环境与生态动力学研究的 Web 模拟平台，将案例创建、研究区域选择、模型文件管理、计算任务执行、实时进度追踪和结果访问整合为统一流程。我主要负责 FVCOM 案例、文件和任务链路：前端使用 React、TypeScript 和 Zustand 管理跨组件状态，通过 SSE 展示长耗时任务进度，并使用虚拟列表优化大量文件的渲染；中台使用 Fastify、Prisma 和 RustFS 完成案例持久化、文件流式传输及计算服务编排。项目中 WebGL 流场渲染由其他成员负责，我负责的是渲染模块上下游的数据和任务接口。

#### A2. selector 为什么能减少渲染？

Zustand 会记录组件通过 selector 选出的值，Store 更新后重新执行 selector，并比较新旧结果。只要选中值没有变化，组件就不需要因 Store 中其他字段改变而重新渲染，因此比订阅整个 Store 更精细。

#### A3. 一次选择多个字段时，如何避免新对象引起重渲染？

如果 selector 每次返回 `{a, b}` 新对象，默认引用比较会认为结果变化。可以分别订阅字段，或者使用 Zustand 的 shallow 比较；更复杂的派生结果可以保证引用稳定或放入 memoized selector。

#### A4. Zustand 状态刷新后为什么消失？

Zustand 默认状态保存在当前 JavaScript 内存中，浏览器刷新会重新创建运行环境。真正需要跨刷新保留的任务状态应以服务端数据库为事实来源；纯前端偏好可使用 `persist` 中间件保存到 localStorage，但不能把它当成可靠的服务端任务状态。

#### A5. 哪些状态更适合服务端状态管理库？

案例详情、文件列表、任务状态等由服务端拥有、需要缓存和重新验证的数据，适合使用 TanStack Query 等工具；弹窗开关、当前滚动位置、地图框选过程等纯交互状态更适合组件 state 或 Zustand。当前项目用 trigger 通知刷新，规模扩大后可以用 query invalidation 收口。

#### A6. EventSource 为什么只适合 GET？如何带鉴权？

浏览器原生 `EventSource` API 只接收 URL 和少量配置，不能像 fetch 一样自由设置方法、body 和请求头。鉴权可以使用同源 Cookie、短期签名查询参数，或改用基于 fetch 流解析的 SSE 客户端；敏感 Token 不应长期暴露在 URL。

#### A7. SSE 必须设置哪些响应头？

至少设置 `Content-Type: text/event-stream`、`Cache-Control: no-cache` 和长连接相关配置。经过 Nginx 时通常还要关闭响应缓冲，并通过心跳避免代理认为连接空闲。

#### A8. `taskRefreshTrigger` 有什么优缺点？

优点是能用递增值表达“又发生了一次刷新”，避免组件直接持有对方方法；缺点是语义较弱、不能携带事件上下文，trigger 增多后难维护。更成熟的方案是由 Store action 收口副作用，或通过服务端状态库的缓存失效机制刷新数据。

#### A9. Lighthouse 中 LCP、TBT、CLS、INP 分别是什么？

- LCP 衡量最大可见内容完成渲染的时间，反映主要内容加载体验。
- TBT 衡量实验室测试中主线程长任务造成的阻塞总时长。
- CLS 衡量页面生命周期内意外布局偏移。
- INP 衡量真实用户交互到下一次绘制的整体响应延迟；Lighthouse 常用 TBT 近似观察交互阻塞，但两者不是同一个指标。

#### A10. 为什么 Lighthouse 会波动？

评分会受 CPU 调度、缓存、网络、第三方资源、浏览器扩展和后台进程影响。因此要固定环境、使用生产构建、清理缓存、多次运行取中位数，并保留报告，而不是只挑最高的一次。

#### A11. 懒加载主要影响哪些指标？

懒加载主要减少首屏需要下载、解析和执行的 JavaScript，可能改善 LCP 和 TBT；如果被延迟的是首屏必需内容，反而可能增加 LCP。虚拟列表更多改善大数据页面的渲染和交互，不一定直接反映在首页 Lighthouse 分数中。

#### A12. Node.js Stream 有哪几类？

主要包括 Readable、Writable、Duplex 和 Transform。上传文件流属于 Readable，对象存储上传端相当于消费它；HTTP 响应是 Writable；压缩流等既读又写并转换数据的属于 Transform。

#### A13. `pipe` 与 `pipeline` 有什么区别？

`pipe` 适合简单连接流，但错误处理和清理需要手动完成；`pipeline` 会统一传播错误并销毁相关流，还可以 Promise 化等待完成。涉及文件落盘或多段流处理时优先使用 `pipeline`。

#### A14. 什么是背压？

当上游生产数据快于下游消费速度时，下游缓冲区可能持续增长。Node Stream 会通过 `write()` 返回值、`drain` 事件及管道机制让上游暂停或恢复，从而限制内存占用。

#### A15. `highWaterMark` 是什么？

它是流内部缓冲区的阈值提示，不是绝对内存上限。达到阈值后可写流通常会返回 false，提示上游等待 `drain`；大小需要在吞吐和内存占用间取舍。

#### A16. 流传输中途失败如何清理？

使用 `pipeline` 统一销毁上下游流，监听请求关闭和上传异常，并删除临时或未完成对象。元数据应只在完整上传成功后切换为有效状态，失败任务还应进入补偿或清理流程。

#### A17. Tree Shaking 与 Code Splitting 有什么区别？

Tree Shaking 是在构建时删除未使用的导出；Code Splitting 是把代码分成多个 chunk 按需加载。前者减少总代码量，后者改变代码的加载时机，两者可以同时使用。

#### A18. 为什么不能只按依赖名配置 `manualChunks`？

手动拆包可能把首屏必需依赖拆成更多请求，也可能制造多个路由共享的大 chunk，甚至破坏执行顺序。应先通过构建分析确认依赖关系，再结合访问路径、缓存命中和 chunk 大小设计，而不是为了消除警告机械拆分。

#### A19. Fastify 为什么性能通常较好？

Fastify 可以基于 JSON Schema 编译参数校验和响应序列化函数，并对请求生命周期做了优化。但本项目主要是 I/O 密集场景，整体性能仍受对象存储、网络和计算服务影响，因此不能只凭框架基准测试声称业务性能提升。

#### A20. TypeScript 为什么不能替代 TypeBox 的运行时校验？

TypeScript 类型在编译后会被擦除，无法约束浏览器或第三方真正发送的数据。TypeBox Schema 可以在运行时校验请求，同时通过 `Static` 推导 TypeScript 类型，从而减少静态类型和运行时规则分离。

### B. React 逐题答案

#### B1. `React.lazy` 为什么期望默认导出？如何兼容具名导出？

`React.lazy` 的约定是 Promise 最终解析为 `{default: Component}`。具名导出可以写成 `import(...).then(m => ({default: m.ModelPage}))`，项目的普通模型页就是这样处理的。

#### B2. `Suspense` 能捕获异步 chunk 加载错误吗？

`Suspense` 只负责等待期间的 fallback，不负责错误展示。动态 chunk 下载失败需要 Error Boundary 捕获；生产环境还可以提供刷新重试，并监控 chunk load error。

#### B3. 为什么副作用使用 `useEffect`，可见范围使用 `useMemo`？

建立 EventSource、监听 ResizeObserver 和操作外部资源属于副作用，需要在 effect 中创建和清理。可见范围是由文件数量、容器高度和滚动位置推导出的纯计算，适合在渲染阶段计算并用 `useMemo` 避免无关渲染时重复执行。

#### B4. `useMemo` 一定能优化性能吗？

不一定。它也需要保存缓存并比较依赖，对很轻的计算可能得不偿失，而且 React 不保证缓存永远存在。应在计算昂贵、引用稳定性重要或 Profiler 证明有效时使用。

#### B5. 滚动位置为什么用 state，而不是 ref？

可见区间直接影响渲染内容，因此滚动位置变化需要触发重新渲染，用 state 更直接。若事件频率过高，可把原始值放 ref，并通过 `requestAnimationFrame` 批量同步到 state。

#### B6. 滚动事件是否需要节流？

当前每次滚动都更新 state，文件多或设备性能较低时可能产生过多渲染。可以使用 `requestAnimationFrame` 节流，使每帧最多更新一次；传统时间节流也可以，但可能降低滚动跟手性。

#### B7. Strict Mode 为什么可能让 effect 执行两次？

React 18 开发环境会模拟挂载、清理、再次挂载，以暴露副作用不幂等和清理不完整的问题。生产环境不会这样重复，但 effect 必须能正确清理 EventSource、Observer 和事件监听。

#### B8. 如何保证 EventSource 不重复创建？

把创建逻辑放入依赖明确的 `useEffect`，cleanup 中关闭旧连接，并避免将每次渲染都会变化的对象放入依赖。若连接应全局唯一，可以封装为单例服务或由顶层任务 Provider 统一管理。

#### B9. 列表为什么使用稳定 key？

稳定 key 帮助 React 识别同一个业务项并复用节点。使用 index 时，删除或重排会让后续元素复用错误状态；文件列表应优先使用对象 key 或文件路径。

#### B10. 删除文件后虚拟列表索引变化有什么风险？

索引之后的条目位置都会前移，如果 key 使用 index，可能出现组件状态错位。使用稳定文件 key，并重新计算占位高度和可见范围；删除末尾数据后还需校正超出新范围的 scrollTop。

#### B11. Zustand 更新后哪些组件会渲染？

订阅结果发生变化的组件会重新渲染。若组件订阅整个 Store，任何字段变化都可能触发；使用字段 selector 和 shallow 比较可以缩小影响范围。

#### B12. `setState` 是同步还是异步？

调用本身会立即进入 React 更新流程，但组件读取到新状态和 DOM 提交不一定同步发生。React 18 会对同一事件循环中的多次更新进行自动批处理，因此不能在 set 后立即假设闭包中的旧变量已经变化。

#### B13. 多个 Store 更新如何避免中间状态不一致？

最好封装成单个领域 action，在一次 `set` 中原子更新相关字段。跨 Store 时可以重构为同一 Store action，或设计明确的状态机，避免组件依赖多个更新的先后顺序。

#### B14. Error Boundary 与请求 `try/catch` 的区别？

`try/catch` 处理 Promise、事件回调等命令式错误；Error Boundary 捕获子组件渲染、生命周期和构造阶段的错误并显示降级 UI。它不会自动捕获异步回调错误，两者需要配合。

#### B15. 页面切换时如何取消请求？

为 fetch 创建 `AbortController`，effect cleanup 时调用 abort，并在 catch 中区分主动取消与真实错误。上传还需要服务端识别连接关闭并清理不完整对象，不能只阻止前端 setState。

### C. 浏览器与网络逐题答案

#### C1. 从输入 URL 到页面展示发生什么？

浏览器解析 URL，查询缓存和 DNS，建立 TCP/TLS 连接，发送 HTTP 请求，解析 HTML；遇到 CSS、JS 等资源继续请求，构建 DOM 和 CSSOM，生成渲染树并完成布局、绘制和合成。模块脚本执行后 React 创建组件树并提交 DOM。

#### C2. 浏览器如何加载 ES Module 和动态 import？

入口 module 按依赖图加载静态 import；遇到 `import()` 时，只有执行到该表达式才请求对应 chunk。chunk 下载、解析和执行完成后 Promise resolve，React.lazy 才能渲染组件。

#### C3. 带 hash 的静态文件为什么适合长期缓存？

内容变化会产生新 hash 和新 URL，因此旧文件可以设置 `Cache-Control: public, max-age=31536000, immutable`。HTML 不应长期强缓存，因为它负责引用最新 hash 文件。

#### C4. HTTP/1.1 下 SSE 有什么限制？

浏览器对同源 HTTP/1.1 并发连接数有限，多个标签页或多个 SSE 会占用连接。HTTP/2 可在一条连接上多路复用多个流，但仍需控制服务端总连接和代理配置。

#### C5. HTTP/2 对资源和 SSE 有什么影响？

HTTP/2 多路复用减少多个资源竞争连接的问题，也能承载长时间响应流。但服务器推送任务语义、鉴权、心跳和断线恢复仍需应用层实现。

#### C6. `AbortController.abort()` 会停止服务端计算吗？

不会必然停止。它只中止浏览器等待和底层请求；服务端可能已经收到请求并启动计算。要真正取消，需要独立取消接口、任务 ID，并由计算服务确认终止。

#### C7. multipart/form-data 的结构是什么？

请求体由 boundary 分隔多个 part，每个 part 有自己的 Content-Disposition、可选 Content-Type 和内容。文件和普通字段可以在同一个请求体中以流式方式解析。

#### C8. CORS 为什么存在？

同源策略限制脚本读取其他源的响应，CORS 由服务端通过响应头声明允许的来源、方法和请求头。生产环境不应无条件允许所有来源，尤其是携带 Cookie 时。

#### C9. `Content-Disposition: attachment` 有什么作用？

它提示浏览器把响应作为下载文件处理，并可通过 filename 指定建议文件名。中文文件名应使用符合 RFC 5987 的 `filename*` 编码。

#### C10. Blob URL、普通 URL、预签名 URL 有什么区别？

Blob URL 指向当前页面内存中的 Blob，生命周期由页面和 `URL.revokeObjectURL` 管理；普通 URL 由服务器长期提供资源；预签名 URL 在有限时间内授予访问特定对象的权限，适合绕过中台直传或直下。

#### C11. 图片为什么影响 LCP？

如果最大内容元素是主视觉图片，其下载、解码和渲染时间直接决定 LCP。应压缩格式、提供正确尺寸、预加载首屏关键图片；首屏 LCP 图片不应使用懒加载。

#### C12. WebP/AVIF 有什么收益？

它们通常能在相近视觉质量下得到比 PNG/JPEG 更小的体积，减少网络时间。AVIF 压缩率更高但编码成本和兼容策略需考虑，可通过 `<picture>` 提供回退。

#### C13. 哪些图片适合 `loading="lazy"`？

首屏之外、用户滚动后才会看到的模型卡片图片适合；首屏主视觉和 LCP 图片不适合。还应设置宽高避免图片加载后造成 CLS。

#### C14. JS 体积为什么影响 TBT？

脚本不仅要下载，还要解析、编译和执行。大型脚本会在主线程形成超过 50ms 的长任务，阻塞输入和绘制，因此 gzip 体积下降不代表执行成本一定同比下降。

### D. SSE 逐题答案

#### D1. SSE 数据格式是什么？

响应是 UTF-8 文本流，每条事件可以包含 `data:`、`event:`、`id:`、`retry:` 等行，并用空行结束。连续多个 `data:` 行会被客户端用换行合并。

#### D2. 为什么每条消息末尾需要两个换行？

单个换行只结束字段行，第二个换行形成空行，表示一条事件结束。没有空行时客户端会继续缓冲，不触发 message。

#### D3. 如何发送自定义事件、ID 和重试间隔？

服务端依次写入 `event: progress`、`id: 123`、`retry: 3000`、`data: {...}`，最后写空行。前端使用 `addEventListener('progress', handler)` 接收。

#### D4. EventSource 默认如何重连？

连接异常关闭后浏览器会按默认或服务端 `retry` 指定的间隔重连，并可能携带最近收到的 `Last-Event-ID`。业务仍需处理重复事件和补发逻辑。

#### D5. 如何补发断线期间事件？

为事件分配单调 ID并持久化；客户端重连携带 Last-Event-ID，服务端从该位置读取并补发。如果没有事件日志，只能重新发送数据库快照恢复最终状态，无法保证每个中间进度都不丢。

#### D6. 为什么先发全量快照？

它让新连接和刷新后的页面立即得到当前事实状态，无需等待下一次变化。之后再接收增量事件，兼顾恢复能力和实时性。

#### D7. 全量和增量如何防止乱序？

为任务更新增加递增版本号或更新时间，客户端只接受版本更大的数据；订阅事件前后需要避免查询快照形成窗口，可先订阅再读取快照，或统一从有序事件日志消费。

#### D8. 数据库成功但事件发送失败怎么办？

数据库仍保存最终事实，客户端重连或定期校准时可恢复。若要求事件可靠，应采用 Outbox：业务事务同时写事件表，再由独立发布器投递。

#### D9. 事件发送成功但数据库失败怎么办？

不应在持久化成功前发布最终事件。当前服务先更新数据库再 emit 是较合理顺序；进一步可以用 Outbox 避免进程在两步之间崩溃。

#### D10. 多浏览器订阅会产生多少 listener？

当前每个 `/progress/tasks` 连接会注册一个 EventEmitter listener，因此连接数与 listener 数大致线性增长。连接关闭时必须移除，否则会泄漏。

#### D11. `setMaxListeners(100)` 解决了什么？

它只提高 Node 对潜在 listener 泄漏的警告阈值，不限制连接、不减少内存，也不证明支持 100 并发。真正容量要通过压测、连接管理和架构设计验证。

#### D12. 多实例下 EventEmitter 为什么失效？

EventEmitter 只在单个 Node 进程内存在。任务更新如果发生在实例 A，而客户端连接实例 B，B 收不到事件；需要 Redis、消息队列或共享事件日志进行跨实例传播。

#### D13. Redis Pub/Sub 如何替代？

任务服务更新状态后向频道发布事件，各 SSE 实例订阅频道并向本机连接转发。Pub/Sub 适合实时广播但不持久化；需要断线补发时应使用 Redis Streams、Kafka 或数据库事件表。

#### D14. Pub/Sub 会丢消息吗？

订阅者离线或网络异常时，普通 Pub/Sub 不保存消息。若要求可靠消费，需要带持久化和消费位点的系统，并结合幂等处理。

#### D15. 如何按用户或 caseID 过滤？

连接建立时校验用户权限并保存订阅集合，收到事件后只向有权限且订阅对应 caseID 的连接发送。规模更大时可以按用户、项目或 caseID 设计频道，减少无关广播。

#### D16. SSE 如何鉴权？

同源系统可使用 HttpOnly Cookie；跨域需正确配置 CORS 与凭证。也可以先请求一次性短期 ticket 再放入查询参数，避免长期 Token 暴露。

#### D17. 为什么代理可能缓冲 SSE？

部分代理为了提高普通响应吞吐，会积累一批内容后再发送，导致事件不能实时到达。需要关闭 proxy buffering、禁止压缩缓冲并定期发送心跳。

#### D18. 如何处理慢客户端？

观察 response 的可写状态和缓冲量，必要时暂停、合并进度事件或断开长期无法消费的连接。进度属于可覆盖状态，可只保留最新值而不是积压全部中间值。

#### D19. 如何限制单用户连接数？

服务端按用户记录活动连接，超过限制时拒绝新连接或关闭旧连接；同时设置总连接上限、空闲超时和监控告警。多实例时计数需要共享存储或网关统一管理。

#### D20. 如何监控 SSE？

记录当前连接数、连接建立/关闭原因、平均持续时间、发送失败、重连次数和事件延迟。前端也应上报 `onerror`、恢复耗时和任务状态长时间不更新。

#### D21. 高频进度如何合并？

服务端可按固定时间窗口只发送最新进度，或只在进度跨过一定阈值时发送。因为进度是状态而不是必须逐条消费的日志，丢弃被更新覆盖的中间值通常可以接受。

### E. 虚拟列表逐题答案

#### E1. overscan 为什么选择 6？

它是流畅度和额外 DOM 数量的折中，不应只凭经验宣称最优。可以在不同滚动速度和设备上测试白屏率、渲染耗时和节点数量，再选择满足体验的最小值。

#### E2. 为什么保留完整占位高度？

滚动条长度由滚动内容总高度决定。即使只渲染少量行，也要用 `itemCount × rowHeight` 模拟完整内容高度，浏览器才能提供正确滚动范围。

#### E3. 绝对定位为什么不撑开容器？

绝对定位元素脱离正常文档流，不参与父元素高度计算。因此需要外层显式设置总高度，再通过 top 把可见行放到对应位置。

#### E4. `floor` 和 `ceil` 用在哪里？

起始索引使用 floor，包含顶部部分可见的行；结束索引使用 ceil，包含底部部分可见的行。然后在两侧加入 overscan 并限制在合法范围。

#### E5. 容器高度为 0 时为什么兜底？

组件首次渲染或隐藏面板时，ResizeObserver 可能还没得到有效尺寸。如果直接计算会渲染空列表，因此先渲染少量条目，尺寸有效后再重新计算。

#### E6. ResizeObserver 有什么性能风险？

尺寸频繁变化可能高频触发回调和重新渲染；回调内读写布局还可能造成抖动。可以只在高度实际变化时更新，并用 requestAnimationFrame 合并。

#### E7. 快速滚动如何避免白屏？

增加适当 overscan、使用 requestAnimationFrame 更新、减少行组件渲染成本，并避免在滚动事件中执行同步重计算。极快滚动时也可以展示轻量占位。

#### E8. 如何滚动到指定索引？

固定行高时设置 `scrollTop = index × rowHeight`；如果要居中或底部对齐，再减去相应视口偏移。不定高列表需要先通过累计高度索引得到目标 offset。

#### E9. 删除前部数据如何保持视觉位置？

记录删除前锚点项及其视口偏移，数据变化后找到同一 key 的新位置并修正 scrollTop。简单固定行高场景也可以按删除行数乘行高补偿。

#### E10. 不定高列表怎么做？

渲染后通过 ResizeObserver 测量每行高度，维护高度缓存和前缀和；用二分查找从 scrollTop 定位起始项。高度变化时更新后续偏移，并保持滚动锚点。

#### E11. 十万条数据时数组会不会成为问题？

虚拟化只减少 DOM，并不减少前端持有的数据和排序过滤成本。十万条时应考虑服务端分页、增量加载、索引结构以及避免每次渲染全量 map/filter。

#### E12. 虚拟化会减少数据请求和内存吗？

不会自动减少。当前文件数组仍全部存在内存中；要减少网络和数据内存，需要分页、窗口化请求或游标加载。

#### E13. 如何用 Profiler 验证？

构造相同数据量，分别记录完整列表与虚拟列表的 mount/commit 时间、滚动时 commit 次数和持续时间，同时在 Elements 面板统计节点数量。测试应固定设备、浏览器和数据。

#### E14. 为什么 Lighthouse 不一定体现虚拟列表收益？

Lighthouse 通常测试初始加载且可能没有打开大文件面板或执行滚动交互。虚拟列表的收益主要出现在大量数据渲染和滚动过程中，更适合用 Profiler、DOM 节点数和交互帧率评估。

#### E15. 如何测试虚拟列表？

单元测试验证不同 scrollTop、容器高度和数据量下的首尾索引；组件测试验证只渲染范围内条目、删除和 ResizeObserver 更新；端到端测试验证快速滚动、滚动到末尾和空列表。

### F. Fastify、Node.js 与数据库逐题答案

#### F1. Fastify 插件与 Express 中间件有什么差异？

Fastify 插件具有封装作用域，插件中的装饰器、Hook 和路由默认不会任意污染外部实例，便于模块化；Express 中间件更接近按注册顺序执行的共享链路。两者都能完成需求，但组织模型不同。

#### F2. TypeBox 如何提供两种类型能力？

TypeBox 使用 JavaScript 对象描述 JSON Schema，Fastify在运行时用它校验；TypeScript 通过 `Static<typeof Schema>` 推导静态类型，使路由代码得到类型提示。

#### F3. Fastify 如何优化校验和序列化？

它会基于 JSON Schema 编译专用校验器和响应序列化函数，避免每次请求执行完全通用的反射逻辑。实际收益还取决于业务 I/O 和下游耗时。

#### F4. 为什么统一响应结构？

前端可以统一处理成功、失败、数据和消息，减少每个接口重复判断。但 HTTP 状态码仍应准确表达协议层结果，统一业务结构不应成为所有错误都返回 200 的理由。

#### F5. 所有错误返回 HTTP 200 有什么问题？

浏览器、网关、监控和重试组件会把它当作成功，缓存和告警也难以区分。更合理的是参数错误 400、未认证 401、无权限 403、不存在 404、冲突 409、服务端错误 500，同时 body 保留统一错误码和消息。

#### F6. 全局错误处理如何改进？

定义可识别的业务错误类型和错误码，区分校验、权限、冲突、外部服务及未知异常；对外返回脱敏信息，对内记录 requestId、堆栈和上下文。未知异常才统一映射为 500。

#### F7. 如何避免泄露敏感信息？

不直接返回 error.message、堆栈、本地路径、对象存储密钥和下游地址；日志也要脱敏 Token 和文件敏感信息。对外只返回稳定错误码、用户可理解信息和追踪 ID。

#### F8. Node 单线程为什么能处理多连接？

JavaScript 回调在单个事件循环线程执行，但网络和文件 I/O 由操作系统及 libuv 协调，完成后再把回调放入事件循环。因此适合 I/O 密集任务，但 CPU 密集计算会阻塞所有请求。

#### F9. 哪些操作会阻塞事件循环？

大型同步循环、同步文件 API、超大 JSON 解析/序列化、加密压缩和直接运行科研计算都会阻塞。应改为异步 I/O、Worker Thread、子进程或独立计算服务。

#### F10. 为什么科研计算不放在 Fastify 主进程？

计算耗时长且 CPU/内存需求高，会阻塞 Web 请求并降低故障隔离。中台只负责任务编排，实际模型由独立计算服务运行，便于独立扩容、限制资源和失败恢复。

#### F11. `execa` 如何管理外部进程？

启动后保存 PID，监听 stdout/stderr 更新进度，等待 Promise 获得退出结果，并根据退出码更新任务状态。还要处理进程启动失败、服务重启后 PID 失效及僵尸进程。

#### F12. 服务重启后内存状态会怎样？

EventEmitter listener、定时器和内存 PID 映射都会丢失，但外部计算可能继续运行。应把任务 ID、计算服务任务 ID和状态持久化，重启后查询计算服务恢复，而不是只依赖进程内变量。

#### F13. 如何用任务队列改造？

执行接口只创建任务并入队，Worker 获取任务、调用计算服务并更新状态，失败按策略重试或进入死信队列。Web 服务和 Worker 解耦后可分别扩容，并提供排队、并发限制和可观测性。

#### F14. Prisma 连接池如何考虑？

连接数应根据数据库上限和应用实例数分配，避免每个实例创建过多连接。SSE 长连接本身不应长期占用数据库连接，只在查询或更新时短暂使用；Serverless 等场景可使用连接代理。

#### F15. 8 位随机 ID 的碰撞概率如何看？

可用空间约九千万，少量案例时碰撞概率低，但随数量按生日悖论增长；当前代码最多重试十次只是降低即时冲突。更稳妥的是数据库唯一约束配合 UUID、雪花 ID 或随机 ID 冲突重试。

#### F16. 为什么 UUID 更适合作主键？

UUID 空间大、分布式生成无需中心协调，冲突概率极低。缺点是字符串较长、随机 UUID 对 B 树局部性不友好；可根据数据库和排序需求选择 UUIDv7 等有时间顺序的方案。

#### F17. 删除案例的正确顺序是什么？

先阻止新任务并确认运行任务取消，再将案例标记为 deleting；异步删除对象和关联数据，全部成功后删除主记录或标记 deleted。采用状态和补偿比一次请求内硬删除所有资源更可靠。

#### F18. 如何让删除接口幂等？

删除不存在或已删除资源时返回可接受结果；使用状态机防止重复执行，并让对象存储删除不存在 key 也视为成功。客户端重试不应造成新的副作用。

### G. 文件与安全逐题答案

#### G1. 流式上传等于分片上传或断点续传吗？

不等于。流式上传只是避免服务端一次性缓存完整文件；分片上传需要把文件拆成独立片段并记录 uploadId 和分片状态，断点续传还要查询已完成分片并恢复。

#### G2. 如何限制文件大小？

在 Fastify multipart 配置 `fileSize` 上限，并处理超限错误；网关和对象存储也要设置限制。前端检查只能改善体验，不能替代服务端校验。

#### G3. 为什么不能相信 filename？

filename 由客户端提供，可能包含路径字符、控制字符、重名或伪造扩展名。应取 basename、生成服务端对象键、限制字符和长度，并把原始显示名与存储 key 分开。

#### G4. 如何防止路径穿越？

不要把未经处理的 filename 直接拼接到本地路径；使用 basename、允许字符白名单，并校验规范化后的路径仍位于目标目录。对象存储 key 也应由服务端生成。

#### G5. MIME 和扩展名可信么？

都可伪造。需要读取文件魔数或使用解析器验证内容；对专业模型文件还可以检查必需字段和格式结构。

#### G6. 文件 hash 有什么作用？

可校验上传完整性、识别重复文件、实现秒传和分片合并校验。大文件 hash 应增量计算，不能先读入完整内存。

#### G7. 同名并发覆盖如何处理？

使用版本化 key 或临时 key，数据库通过版本号/条件更新决定哪个上传生效；失败或失效对象异步清理。单纯先删旧对象再上传存在竞争和空窗。

#### G8. 为什么按 caseID/input/output 分层？

它提供业务隔离和清晰生命周期，便于按案例列举、授权、清理输入和结果，也让计算服务只需约定前缀而不关心数据库结构。

#### G9. 如何防止任意 key 越权下载？

不要只检查 key 非空；应根据当前用户和 caseID 查询权限，再验证 key 确实属于该案例。也可以生成短期预签名 URL，且对象存储 bucket 默认私有。

#### G10. 公共 URL 有什么风险？

任何拿到 URL 的人可能绕过应用鉴权长期访问，且难以撤销。敏感结果应使用私有 bucket、中台鉴权下载或短期预签名 URL。

#### G11. 什么时候使用预签名 URL？

大文件直传/直下、希望减轻 Node 带宽压力且权限可以短期表达时使用。签名要限制对象 key、方法、有效期和可选内容类型。

#### G12. 直传对象存储有什么优缺点？

优点是减少中台带宽、内存和连接占用；缺点是需要额外签名、CORS、上传完成回调和安全校验。业务元数据仍要由中台确认，不能仅凭客户端说上传成功。

#### G13. 如何实现分片上传？

服务端创建上传会话，前端按大小切片并并发上传，记录 ETag；完成后请求对象存储合并。恢复时查询已上传分片，只传缺失部分，最终校验文件 hash。

#### G14. 上传后如何扫描文件？

文件先进入隔离区并标记 scanning，异步 Worker 做病毒扫描、格式校验和业务解析；通过后移动或标记 valid，失败则隔离/删除且不允许下载或计算服务消费。

#### G15. 中文下载文件名如何处理？

同时提供 ASCII fallback 的 `filename` 和 UTF-8 编码的 `filename*=UTF-8''...`，避免不同浏览器乱码，并清理换行等可能造成响应头注入的字符。

#### G16. 如何支持 Range 请求？

解析 `Range` 请求头，从对象存储读取对应字节区间，返回 206、`Content-Range` 和 `Accept-Ranges`。无效范围返回 416。

#### G17. Stream 失败如何删除不完整对象？

使用对象存储 multipart upload 时主动 abort 上传会话；普通流式上传则使用临时 key，成功后再提升为正式对象，失败对象由清理任务删除。

#### G18. RustFS 不可用如何降级？

上传和下载属于核心能力，通常不能伪造成功；应快速失败并返回可重试错误，使用熔断和监控保护中台。读取场景可按业务需要使用缓存，但不能以过期文件冒充最新结果。

#### G19. 如何统计上传进度？

浏览器可使用 XHR/axios 的 upload progress，或自行分片统计已上传字节；fetch 对上传进度支持有限。服务端也可记录接收字节，但要建立独立通道反馈给前端。

#### G20. 如何做对象生命周期管理？

对象元数据记录创建时间、状态和引用关系；对临时上传、失败任务和过期案例设置生命周期规则或定时清理，并在删除前确认无有效引用。

### H. 工程化逐题答案

#### H1. 为什么使用 Vite？

开发环境利用原生 ESM 按需转换模块，启动和热更新通常较快；生产使用 Rollup 打包。项目使用 React SWC 插件进一步提升 JSX/TS 转换速度。

#### H2. Vite 开发和生产机制有什么不同？

开发时不打完整 bundle，浏览器按 ESM 请求模块，Vite按需转换并进行依赖预构建；生产时 Rollup 做打包、Tree Shaking、拆包、压缩和 hash 命名。

#### H3. SWC 承担什么作用？

SWC 使用 Rust 实现，负责快速转换 TypeScript/JSX 和 React Refresh 相关处理。它主要提升编译体验，不替代 TypeScript 的完整类型检查。

#### H4. 为什么出现 500 kB chunk 警告？

说明压缩前某些 chunk 较大，可能包含 Mapbox、ECharts、Ant Design 或多个业务模块。警告本身不是错误，但提示下载、解析和执行成本可能影响性能，需要先分析组成。

#### H5. `manualChunks` 按依赖还是业务拆？

没有固定答案。稳定且多路由共享的大依赖可拆 vendor 便于缓存，路由专属代码按业务拆；同时要避免过度碎片化和一个巨大的公共 vendor 包。

#### H6. 拆得过细有什么问题？

会增加请求、调度和模块执行开销，产生依赖瀑布，并降低压缩率；共享依赖变化还可能导致更多 chunk hash 失效。HTTP/2 能缓解连接问题，但不能消除解析执行成本。

#### H7. 如何定位入口包过大？

使用 Rollup visualizer 或 source-map-explorer 查看模块占比，再结合 Network 面板确认首屏实际加载。项目当前还应检查 Router 对地图 Store 的静态引用和首页重依赖。

#### H8. ESLint、Prettier 和类型检查分别做什么？

ESLint发现代码模式和潜在错误，Prettier统一格式，`tsc --noEmit` 做类型检查。当前 build 是否完整执行类型检查需要确认，不能把 SWC 转译成功等同于类型正确。

#### H9. 当前测试覆盖的不足是什么？

仓库测试较少，核心的 SSE 生命周期、虚拟列表边界、文件上传失败和任务状态机缺少系统测试。应优先覆盖高风险状态转换和资源清理，而不是只追求覆盖率数字。

#### H10. SSE 如何测试？

单元测试事件序列解析和状态归并；服务端集成测试验证响应头、首次快照、状态事件及断开后 listener 清理；端到端测试模拟断线重连和任务完成。

#### H11. 文件上传如何测试？

覆盖合法文件、超限、空文件、同名替换、并发覆盖、对象存储失败和客户端中断；验证对象、数据库元数据和临时资源最终一致。

#### H12. 如何在 CI 中稳定运行 Lighthouse？

固定生产构建、Chrome/Lighthouse 版本、页面数据和网络节流，在独立 Runner 上多次运行取中位数。外部地图等不稳定资源应使用稳定测试环境或明确其影响。

#### H13. Lighthouse CI 如何设置预算？

对 Performance 分数、LCP、TBT、CLS、总 JS 体积和单 chunk 大小设置最低/最高阈值，PR 超出预算时告警或阻断。阈值应基于可靠基线逐步收紧。

#### H14. 如何监控真实用户性能？

在生产页面采集 Web Vitals，如 LCP、CLS、INP，并附带页面、设备、网络和版本信息上报。使用分位数而非平均值观察真实用户体验。

#### H15. 实验室数据和 RUM 有什么区别？

Lighthouse 是可控环境下的合成测试，适合回归和定位；RUM 来自真实用户，能反映设备、网络和交互差异，但噪声更大。二者应互补。

#### H16. 如何设置缓存与回滚？

hash 静态资源长期缓存，HTML 短缓存或协商缓存；部署保留上一版本完整产物，回滚时切换 HTML/版本入口。不能只回滚 HTML 却删除它引用的旧 chunk。

### I. 架构设计补充答案

#### I1. 千级任务下前端如何优化？

SSE 改为按用户增量推送，只更新发生变化的任务；任务面板使用虚拟列表，避免每次事件全量排序和创建新对象。服务端使用消息通道、队列和批量/节流进度落库。

#### I2. 多实例为什么可能需要粘性会话？

如果连接相关订阅只保存在单实例内，重连到其他实例需要重新建立订阅；不过使用共享事件通道和无状态鉴权后可以不依赖粘性会话。SSE 本身建立后会固定在一个实例直到断开。

#### I3. 可靠任务系统为什么需要任务 ID，而不是只用 caseID？

同一案例可能多次执行。独立 taskID 能区分不同运行批次、支持重试和审计，并防止旧任务的完成事件覆盖新任务状态。

#### I4. 如何防止重复执行？

客户端 disabled 只能改善体验，服务端需要基于 caseID、任务版本或幂等键做唯一约束/条件更新。只有从非 running 状态原子切换成功的请求才能创建任务。

#### I5. 如何防止旧事件覆盖新任务？

事件携带 taskID 和递增版本号，前端和数据库只接受当前任务且版本更大的更新。仅凭 caseID 和 progress 无法可靠区分两轮运行。

> 我负责案例管理、文件链路、任务状态和 Node 中台接口；与地图模块协作确定区域 bounds 和结果资源格式。WebGL/GLSL 的渲染算法及自定义图层内部实现由其他成员完成。

> 需求来自水环境科研模拟场景，主要使用者是需要管理模型输入、启动计算和查看结果的科研人员。具体来源和用户数量应按真实情况补充，不要虚构。

> 可讲真实的“任务取消后按钮或任务面板没有同步刷新”问题：根因是当前案例状态和任务关注状态耦合，修复时将二者拆开，并通过服务端状态和刷新信号联动。若这不是你的真实排障经历，请换成实际案例。

#### J5. SSE 是自己选择还是指定？

> 我先根据单向、低频任务进度的特点比较轮询、WebSocket 和 SSE，再选择 SSE。重点不是技术新颖，而是通信方向和复杂度匹配。

#### J6. 哪个设计最初不好？

> 可以讲 trigger 字段逐渐增多：早期能快速解耦组件，但规模扩大后语义分散；如果重构会用统一任务 action 或服务端缓存失效收口。这个回答既说明取舍，也承认演进空间。

#### J7. 如何定位取消状态 Bug？

> 先复现并记录前端 Store、接口响应和数据库状态，判断是事件未到达、Store 未更新还是计算服务未取消；再沿“操作请求—服务端状态—SSE 事件—前端归并”链路逐层定位，而不是只修改按钮显示。

#### J8. 做过哪些技术取舍？

> 可以讲 SSE 与 WebSocket、Zustand 与 Redux、流式上传与完整 Buffer、固定行高虚拟列表与通用不定高方案。回答要说明业务约束、采用方案和放弃方案的成本。

#### J9. 时间更紧时保留什么？

> 优先保留案例、文件、执行和可靠状态恢复这条核心链路；虚拟列表和复杂动画可在确认数据规模后迭代。安全校验、错误处理和数据一致性不能作为“体验优化”随意删除。

#### J10. 如何证明虚拟列表不是过度设计？

> 用真实或模拟文件规模测完整渲染的节点数、Commit 耗时和滚动表现。若实际案例始终只有十几份文件，虚拟列表收益有限，应诚实承认；若达到数百或更多，再用数据证明必要性。

#### J11. 收到过什么代码评审意见？

应准备一个真实例子，结构为“原方案—评审指出的问题—修改—学到什么”。没有记录就不要编造具体人物或结论。

#### J12. 最有价值的重构是什么？

可优先讲把当前案例与任务关注状态拆开，或将轮询任务面板改为统一 SSE 通道；但必须与你实际提交一致。

#### J13. 如何判断性能问题值得优化？

先用构建报告、Lighthouse、Profiler 或真实用户数据建立基线，再评估影响用户范围和优化成本。当前入口约 1.34 MB gzip 是明确证据；不能因为“感觉慢”就宣称优化成功。

#### J14. 哪项独立完成，哪项协作完成？

按真实情况列清楚：例如案例/文件/任务 UI 和中台接口由你负责，地图区域选择需要协作，WebGL 渲染由其他成员负责。用“我负责”和“我们协作”的主语区分。

#### J15. 最大技术成长是什么？

> 我最大的成长是从单个页面功能扩展到完整任务链路，开始把浏览器状态、数据库状态、对象存储和计算服务作为一个系统考虑；也认识到 SSE、重试和对象存储接入只是第一步，可靠性还需要幂等、补偿、事件持久化和可观测性。

## 十五、面试回答通用结构

面对任何项目问题，可以用以下四步：

1. **先说场景和约束**：为什么这个问题在本项目中存在。
2. **再说当前实现**：使用了哪些机制，数据如何流动。
3. **说明实际收益**：降低了什么复杂度或避免了什么风险；有数据才报数字。
4. **主动给出边界**：当前方案在哪些规模或异常情况下不足，如何改进。

示例：

> 任务计算时间较长，而且主要是服务端向前端单向推送，所以我选择 SSE，而不是让每个任务持续轮询。连接建立后服务端先推送数据库快照，之后通过事件总线在状态变化时推送；前端在组件卸载时关闭连接，服务端也移除 listener。当前方案适合单实例和低频进度，如果多实例部署，我会将进程内 EventEmitter 替换为 Redis Streams 或消息队列，并通过事件 ID 支持断线补发。
