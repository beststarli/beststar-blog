## 鉴权设计
### [双token无感刷新](/blog/tech/双token无感刷新的实现.md)如果refresh token也失效了，但这时挂起了一堆请求，刷新失败了，怎么处理?
:::info **思路：请求队列 + 失败重定向**
当 refresh token 也失效时，本质是登录态彻底过期。这时候挂起的请求再刷新也没意义了，所以直接拒绝这些挂起的请求，并跳转到登录页。
:::
#### 1. 请求挂起队列
在发起 refresh 请求时，将这段时间内所有并发请求拦截下来，**不直接拒绝也不放行**，暂存入一个队列：
```ts
let isRefreshing = false
let pendingQueue: Array<{ resolve: Function; reject: Function }> = []

axios.interceptors.response.use(
  response => response,
  async error => {
    const { config, response } = error
    if (response?.status !== 401 || config._retry) return Promise.reject(error)

    // 已有 refresh 在进行 → 挂起
    if (isRefreshing) {
      return new Promise((resolve, reject) => {
        pendingQueue.push({ resolve, reject })
      })
    }

    config._retry = true
    isRefreshing = true

    try {
      const { token } = await axios.post('/auth/refresh', {
        refreshToken: getRefreshToken(),
      })
      setAccessToken(token)

      // 刷新成功 → 重放挂起的请求
      pendingQueue.forEach(({ resolve }) => resolve(axios(config)))
      pendingQueue = []
      return axios(config)
    } catch {
      // refresh 也失败 → 拒绝所有挂起请求
      pendingQueue.forEach(({ reject }) => reject(error))
      pendingQueue = []
      window.location.href = '/login'  // 跳登录页
      return Promise.reject(error)
    } finally {
      isRefreshing = false
    }
  }
)
```

#### 2. 执行流程图
```
请求 A (401)
  ↓
发现 refresh token 存在 → 发起 refresh
  ↓
请求 B (401)          ← A 的 refresh 还没返回
  ↓
isRefreshing = true → 挂起 B
  ↓
请求 C (401)
  ↓
挂起 C
  ↓
refresh 返回
  ├── 成功 → 更新 token → 依次重放 A/B/C
  └── 失败 → 清空队列 → 跳转 /login
```

#### 3. 边界处理
| 场景 | 做法 |
|---|---|
| refresh 失败但不是 401（如网络断开） | 不要跳登录，让用户手动刷新重试 |
| 多 tab 同步 | 用 `localStorage` + `broadcastChannel` 同步刷新状态 |
| 登录/注册接口 401 | 配置 `_skipAuth: true` 白名单，不触发刷新 |

### [JWT登录机制的原理](/docs/Web开发与安全/JWT身份认证.md)
#### JWT结构
`header.payload.signature`，三部分都是 Base64 编码，合起来是一段字符串。

#### 登录流程
```txt
用户登录 → 服务端验证账号密码 → 签发 JWT（含 userId、过期时间） → 返回前端
前端收到 JWT → 存 localStorage → 每次请求带在 Authorization header 里
服务端收到请求 → 验证 JWT 签名 → 解析 payload → 放行或拒绝
```

#### 为什么JWT可以无状态
JWT的签名由服务端密钥生成，只要密钥不泄露，服务端不需要存session，解析JWT就能确认身份。

#### access + refresh双token机制
| | access token | refresh token |
|---|---|---|
| 有效期 | 短（15min~1h） | 长（7天~30天） |
| 存在哪 | 前端内存或 localStorage | localStorage |
| 用途 | 请求鉴权 | 刷新 access token |
| 风险 | 泄露了影响范围小 | 泄露了要立即撤销 |

**为什么不用单 token**：单 token 有效期短则频繁登录体验差，有效期长则泄露风险大。双 token 用短期 access token 保证安全，用长期 refresh token 保证体验。

### 为什么使用JWT? 相比传统session方案有什么优势?
| | Session | JWT |
|---|---|---|
| 存储位置 | 服务端内存/Redis | 客户端持有，服务端不存 |
| 扩展性 | 多台服务器需共享 session（Redis） | 天然支持水平扩展，任何服务器都能验 |
| 请求开销 | 每次查 session（可能查 DB/Redis） | 直接验签名，无需查存储 |
| 跨域 | 难处理（Cookie 跨域受限） | 放 Authorization header，天然跨域 |
| 实时撤销 | 服务端删除 session 即可 | 需维护黑名单或缩短有效期 |

- **JWT 核心优势**：**无状态**。服务端不需要存任何登录信息，验签名就能确认身份，多机部署不用共享 session。
- **JWT 核心劣势**：**不可撤销**。签出去的 token 在到期前永远有效，无法主动踢人，只能等过期或维护黑名单。

**总结**：JWT 无状态，服务器不存 session，直接验签名即可确认身份，天然支持水平扩展。代价是签发后不可撤销，只能等过期。

### 登录怎么实现? 密码怎么传给后端? 前端加密后后端直接存数据库吗?
#### 登录流程
```
用户输入密码 → HTTPS 传输 → 后端接收明文 → bcrypt 哈希 → 存 hash 到数据库
                                          （不是存密码原文）
```

#### 关键点
| 环节 | 正确做法 | 常见误区 |
|---|---|---|
| 传输 | **HTTPS** 保证传输安全 | 前端加密 MD5 传过去（等于明文，MD5 可查表） |
| 存储 | 后端 **bcrypt/scrypt** 加盐哈希后存数据库 | 直接存密码原文或只 MD5 一次 |
| 验证 | 取出 hash，用 bcrypt 比对 | 解密密码原文比较 |

- **前端要不要加密**：不需要。HTTPS 已经加密传输了。前端加密反而没用（加密逻辑暴露在浏览器，知道用什么算法就能还原）。
- **为什么要用 bcrypt 而不是 MD5**：MD5 是快速哈希，算得快 = 撞库快。bcrypt 可以调节计算成本（工作因子），故意让它慢，拖慢暴力破解。

### 为什么不用单 token，让后端去刷新?
**单 token 方案的问题**：
| | 单 token（短过期） | 单 token（长过期） | 双 token |
|---|---|---|---|
| 过期时间 | 15 分钟 | 7 天 | access 短 + refresh 长 |
| 用户体验 | 每 15 分钟要重新登录 | 好 | 自动续期，无感知 |
| 安全风险 | 低 | 高（token 泄露=永久通行） | 中（refresh 泄露可刷新，但有效期可限制） |

**单 token 无法同时满足安全和体验**：时间短则频繁登录烦人，时间长则泄露风险大。双 token 用短期的 access token 减少泄露影响，用长期的 refresh token 保体验，且 refresh token 可以在服务端维护黑名单实现撤销。

## 微前端
### 集成一个第三方的摄像监控系统，用iframe嵌进去，而且还要求根据用户权限控制展示内容，怎么处理?
:::info **核心矛盾**
iframe 跨域隔离 → 无法直接操作第三方内容 + 需要按权限动态展示。
:::
#### 方案一：URL token + 第三方配合（推荐）
第三方提供接口，前端通过 URL 参数传递用户权限标识：
```ts
// 前端：根据权限生成带 token 的 URL
const token = generateJwt({ userId, role, cameras: user.cameraIds })
setIframeSrc(`https://监控系统.com/embed?token=${token}`)
```
第三方在 iframe 内自行解析 token，只渲染有权限的摄像头。

#### 方案二：postMessage 动态控制
第三方支持 postMessage 协议，前端按权限动态发指令：
```ts
const iframe = document.querySelector('iframe')

function updatePermissions(cameraIds: string[]) {
  iframe.contentWindow?.postMessage(
    { type: 'update-cameras', cameraIds },
    'https://监控系统.com'
  )
}

window.addEventListener('message', (e) => {
  if (e.origin !== 'https://监控系统.com') return
  console.log('iframe 状态:', e.data)
})
```

#### 方案三：多 iframe + 条件渲染
第三方不配合时的兜底方案：
```tsx
function CameraPanel({ user }: { user: User }) {
  const availableCameras = CAMERAS.filter(c => user.cameraIds.includes(c.id))
  return (
    <div className="grid grid-cols-2 gap-4">
      {availableCameras.map(cam => (
        <iframe key={cam.id} src={`https://监控系统.com/camera/${cam.id}`} />
      ))}
    </div>
  )
}
```

#### 方案四：服务端代理转发
后端做权限校验，代理请求第三方视频流，前端只负责展示。

#### 总结
| 方案 | 可控性 | 依赖第三方 | 推荐场景 |
|---|---|---|---|
| URL token | 高 | 需配合 | **首选** |
| postMessage | 中 | 需支持协议 | 已有 postMessage 接口 |
| 多 iframe | 低 | 不需要 | 兜底 |
| 服务端代理 | 最高 | 需提供 API | 安全性要求高 |


## 数据结构
### 如果你从多个系统的API拉数据，但这些系统的数据结构不一致，比如A系统返回camelCase、B系统是snake_case，还有一些是中文key，怎么统一数据结构来复用?

:::info **核心思路**
防腐层（Anti-Corruption Layer）+ 统一 Schema

**API防腐层**：对外部系统的数据结构做一层适配，转换成内部统一的标准结构。上层业务只认标准结构，不关心外部系统的差异。也就是*在外部系统和业务代码之间加一层隔离，防止外部系统的"脏东西"污染核心代码。*
:::
内部定义一套标准结构（推荐 camelCase），给每个外部系统写一个适配器做转换，上层业务只认标准结构。

#### 1. 定义标准 Schema
```ts
// 内部统一数据结构
interface User {
  userId: string
  userName: string
  createdAt: string
}
```

#### 2. 按数据源写适配器
```ts
// A 系统：camelCase → 直接映射
function adaptFromSystemA(raw: any): User {
  return {
    userId: raw.userId,
    userName: raw.userName,
    createdAt: raw.createdAt,
  }
}

// B 系统：snake_case → 转换键名
function adaptFromSystemB(raw: any): User {
  return {
    userId: raw.user_id,
    userName: raw.user_name,
    createdAt: raw.created_at,
  }
}

// C 系统：中文 key → 映射
function adaptFromSystemC(raw: any): User {
  return {
    userId: raw['用户ID'],
    userName: raw['用户名'],
    createdAt: raw['创建时间'],
  }
}
```

#### 3. 统一调用入口
```ts
const adapters = {
  systemA: adaptFromSystemA,
  systemB: adaptFromSystemB,
  systemC: adaptFromSystemC,
}

async function fetchUsers(source: keyof typeof adapters) {
  const raw = await apiMap[source].getUsers()
  return adapters[source](raw)  // 返回统一结构
}
```

#### 4. 通用工具：递归转 camelCase
```ts
import { camelCase, mapKeys, mapValues } from 'lodash-es'

function toCamelCase(obj: any): any {
  if (Array.isArray(obj)) return obj.map(toCamelCase)
  if (obj !== null && typeof obj === 'object') {
    return mapKeys(
      mapValues(obj, toCamelCase),
      (_, key) => camelCase(key),
    )
  }
  return obj
}
```

#### 总结
```
外部数据 → 适配器(Adapter) → 统一 Schema → 业务层
  ↑              ↑               ↑
蛇形/中文/...   各自写一份       只认一套
```

**一句话**：对内定义标准接口，对外用适配器做转换，不改上游、不改下游、只加一层。

## 组件设计
### 如果项目有20+个页面，需求方经常改字段、加字段，每次都要改组件，怎么设计?
:::info **核心思路**
配置驱动 + 递归渲染组件
:::
把页面描述为 JSON 配置，组件根据配置自动渲染，改字段只需要改配置，不用改组件。

#### 1. 定义配置结构
```ts
// 描述一个表单/详情页的配置
interface FieldConfig {
  key: string
  label: string
  type: 'input' | 'select' | 'date' | 'image' | 'table'
  required?: boolean
  hidden?: boolean
  options?: { label: string; value: string }[]
  children?: FieldConfig[]  // 嵌套子表
  render?: (value: any, record: any) => ReactNode  // 自定义渲染
}
```

#### 2. 递归渲染组件
```tsx
function DynamicField({ field, value, onChange }: Props) {
  if (field.hidden) return null

  switch (field.type) {
    case 'input':
      return <Input label={field.label} value={value} onChange={onChange} />
    case 'select':
      return <Select label={field.label} options={field.options} value={value} onChange={onChange} />
    case 'table':
      return (
        <SubTable config={field.children} value={value} onChange={onChange} />
      )
    case 'image':
      return <ImageUpload label={field.label} value={value} onChange={onChange} />
    default:
      return field.render?.(value, { field })
  }
}
```

#### 3. 页面由配置驱动
```ts
// 需求改字段 → 只改这个配置对象
const userFormConfig: FieldConfig[] = [
  { key: 'name', label: '姓名', type: 'input', required: true },
  { key: 'gender', label: '性别', type: 'select', options: [...] },
  { key: 'avatar', label: '头像', type: 'image' },
  { key: 'education', label: '教育经历', type: 'table', children: [...] },
]

function UserPage() {
  const [form, setForm] = useState({})
  return (
    <Form>
      {userFormConfig.map(field => (
        <DynamicField
          key={field.key}
          field={field}
          value={form[field.key]}
          onChange={v => setForm(prev => ({ ...prev, [field.key]: v }))}
        />
      ))}
    </Form>
  )
}
```

#### 4. 加字段流程
```
旧流程：需求加字段 → 找页面 → 改 JSX → 改类型 → 改接口 → 改验证
新流程：需求加字段 → 在配置数组加一项
```

#### 5. 进阶：服务端下发配置
如果字段改动非常频繁，配置甚至可以不下发代码，由后端接口返回：
```ts
// 接口同时返回数据和页面配置
const { data, fields } = await api.get('/page/user')
// data 是表单数据，fields 是渲染配置
return <PageRenderer fields={fields} data={data} />
```

#### 总结
:::success **面试要点**
不要说"用配置驱动"，要说出具体怎么做的——*JSON Schema 描述页面、递归组件渲染、配置与服务分离、变更只改一处。*
:::

### 如果要做一个拖拽式仪表盘Dashboard，用户能自己拖拽、组合图表组件展示不同数据源的信息，怎么设计?
:::info **核心思路**
栅格布局 + 组件注册机制 + 配置持久化
:::

#### 1. 整体架构
```
组件库 (Widget Registry) ← 注册 → 拖拽面板 (Dashboard Editor)
       ↑                              ↓
  内置图表 / 自定义组件         生成 JSON 配置
                                    ↓
                              配置持久化 (保存到后端)
                                    ↓
                             渲染器 (Dashboard Renderer)
```

#### 2. 组件注册机制
每种图表/组件预先注册到系统中：
```ts
interface WidgetMeta {
  type: string          // 唯一标识，如 'line-chart'
  name: string          // 显示名，如 '折线图'
  component: React.ComponentType<WidgetProps>
  defaultW: number      // 默认宽 (栅格单位)
  defaultH: number      // 默认高
  configurable?: FieldConfig[]  // 可配置属性
}
```

#### 3. 布局数据结构
```ts
interface DashboardConfig {
  id: string
  name: string
  widgets: WidgetInstance[]
}

interface WidgetInstance {
  id: string
  type: string          // 组件类型，从注册表取
  x: number
  y: number
  w: number
  h: number
  props: Record<string, any>
  dataSource?: {
    api: string
    params: Record<string, any>
  }
}
```

#### 4. 关键库选型
| 需求 | 推荐库 | 说明 |
|---|---|---|
| 拖拽 + 栅格布局 | `react-grid-layout` | 支持拖拽、缩放、响应式 |
| 图表渲染 | `echarts` / `recharts` | 按需选一个 |
| 拖拽排序 | `dnd-kit` | 侧边栏组件列表拖拽 |

#### 5. 渲染器实现
```tsx
function DashboardRenderer({ config }: { config: DashboardConfig }) {
  return (
    <GridLayout
      layout={config.widgets.map(w => ({
        i: w.id, x: w.x, y: w.y, w: w.w, h: w.h,
      }))}
      cols={12}
      rowHeight={60}
    >
      {config.widgets.map(widget => {
        const meta = widgetRegistry[widget.type]
        if (!meta) return null
        return (
          <div key={widget.id}>
            <meta.component {...widget.props} dataSource={widget.dataSource} />
          </div>
        )
      })}
    </GridLayout>
  )
}
```

#### 6. 编辑模式 vs 查看模式
```tsx
function Dashboard({ mode }: { mode: 'edit' | 'view' }) {
  if (mode === 'edit') return <DashboardEditor />  // 可拖拽
  return <DashboardRenderer />                       // 只展示
}
```

#### 7. 数据源管理
每张图表绑定不同数据源，统一适配：
```ts
async function fetchWidgetData(source: WidgetInstance['dataSource']) {
  const raw = await fetch(source.api, { params: source.params })
  return normalizeData(raw)  // 统一成 { labels: [], values: [] }
}
```

#### 总结
:::success **面试要点**
不要只说"用 react-grid-layout"，而是讲清楚*组件注册机制、配置与渲染分离、数据源抽象、编辑/查看双模式。*
:::

## 性能优化
### 如果现在一个页面要同时展示300辆车的实时位置+轨迹，怎么优化?
:::info **核心瓶颈**
300 辆车 × 实时更新 × 轨迹点 → DOM 爆炸 + 渲染卡顿
:::

#### 方案一：Canvas 渲染替代 DOM（首选）
DOM 元素超过几百个就会卡，Canvas 是位图绘制，几千个点无压力：
```tsx
// 使用 Canvas 覆盖在地图上绘制车辆和轨迹
function VehicleLayer({ vehicles }: { vehicles: Vehicle[] }) {
  const canvasRef = useRef<HTMLCanvasElement>(null)

  useEffect(() => {
    const ctx = canvasRef.current?.getContext('2d')
    if (!ctx) return

    // 清空画布
    ctx.clearRect(0, 0, ctx.canvas.width, ctx.canvas.height)

    // 批量绘制轨迹 + 车辆图标，避免频繁 DOM 操作
    vehicles.forEach(v => {
      // 画轨迹线
      ctx.beginPath()
      ctx.strokeStyle = v.color
      ctx.lineWidth = 2
      v.trail.forEach((p, i) => i === 0 ? ctx.moveTo(p.x, p.y) : ctx.lineTo(p.x, p.y))
      ctx.stroke()

      // 画车辆图标（简化为圆形）
      ctx.beginPath()
      ctx.arc(v.x, v.y, 6, 0, Math.PI * 2)
      ctx.fillStyle = v.color
      ctx.fill()
    })
  }, [vehicles])

  return <canvas ref={canvasRef} className="absolute inset-0" />
}
```

#### 方案二：数据层面的优化
```ts
// 1. 只传增量数据，不传全量
ws.onmessage = (e) => {
  const diff = JSON.parse(e.data)  // 只包含位置变化的车辆

// 2. 降低精度：后端返回坐标保留 4 位小数（≈11m 精度）就够用
//    原始: 113.123456789 → 传输 15 字节
//    优化: 113.1234       → 传输 8 字节，省一半

//    1 秒一个点 → 300 车 × 60 点/分 × 每分钟刷新 = 18000 点
//    5 秒一个点 → 300 车 × 12 点/分 = 3600 点，减少 80%
```


#### 方案四：Web Worker 分担计算
把坐标转换、聚合计算放到 Worker 线程，不阻塞主线程渲染：
```ts
// worker.ts
self.onmessage = ({ data }) => {
  const result = clusterVehicles(data.vehicles, data.zoom)
  self.postMessage(result)
}

// 主线程
const worker = new Worker(new URL('./worker.ts', import.meta.url))
worker.postMessage({ vehicles, zoom })
worker.onmessage = ({ data }) => setClustered(data)
```

#### 方案对比
| 维度 | DOM 方案 | Canvas 方案 | WebGL 方案 |
|---|---|---|---|
| 渲染数量 | &lt; 500 | 5000+ | 10万+ |
| 交互（点击/悬浮） | 天然支持 | 需做命中检测 | 复杂 |
| 实现复杂度 | 低 | 中 | 高 |
| 推荐场景 | 简单图表 | **300 辆车最合适** | 上亿点云 |

#### 总结
:::success **面试要点**
不要只说"用 Canvas"，而是要分层讲：*数据层（增量更新+精度压缩）、渲染层（Canvas替代DOM）、可视层（视口裁剪+聚合）、计算层（Worker 分担）。*
:::

### 重排和重绘是怎么引发的? 怎么解决?
- **重排（Reflow）**：改变元素尺寸/位置/布局属性时，浏览器重新计算布局树。
- **重绘（Repaint）**：改变颜色/背景等不影响布局的属性时，浏览器重新绘制。

#### 引发重排的操作
| 操作 | 示例 |
|---|---|
| 修改几何属性 | `width`、`height`、`padding`、`margin` |
| 修改位置 | `top`、`left`、`display: none`、浮动 |
| 读写布局属性 | `offsetWidth`、`clientHeight`、`getComputedStyle` |
| 增删 DOM | `appendChild`、`removeChild` |
| 窗口缩放 | `resize` 事件 |

#### 重绘不引发重排
`color`、`background-color`、`visibility`、`opacity`、`box-shadow`。

#### 优化方案
| 问题 | 解决 |
|---|---|
| 频繁改样式 | 合并用 `classList` 一次性修改，或 `cssText` 批量赋值 |
| 频繁读布局属性 | 先用变量存起来再操作（避免强制同步布局） |
| 批量 DOM 操作 | 用 `DocumentFragment` 或 `display: none` 批量改完再显示 |
| 动画卡顿 | 用 `transform` + `opacity` 做动画（GPU 合成，不触发重排） |
| 复杂动画 | 让元素脱离文档流（`position: fixed/absolute`），不影响其他元素 |

#### 总结
重排必重绘，重绘不一定重排。优化核心是减少布局属性读写、合并样式操作、用 transform 替代 top/left。

### requestAnimationFrame怎么用?
#### 作用
在下次浏览器重绘之前执行回调，常用于 JS 动画。比 `setInterval` 更流畅、更省资源。

#### 用法
```js
function animate() {
  element.style.transform = `translateX(${pos}px)`
  pos += 2
  requestAnimationFrame(animate)  // 递归调用
}
requestAnimationFrame(animate)
```

#### 与setInterval相比
| | requestAnimationFrame | setInterval |
|---|---|---|
| 执行时机 | 跟随浏览器绘制帧（约 16ms） | 固定间隔（不管帧是否空闲） |
| 页面不可见时 | **自动暂停**，不浪费资源 | 继续执行，浪费 CPU |
| 帧率匹配 | 自动匹配屏幕刷新率（60/120Hz） | 固定间隔，可能掉帧或过度绘制 |
| 电池 | 省电 | 费电 |

#### 总结
做动画用 `requestAnimationFrame`，浏览器会自动优化暂停、匹配刷新率。`setInterval` 适合轮询等非动画场景。

### 图片加载优化策略
| 策略 | 做法 | 效果 |
|---|---|---|
| **懒加载** | `<img loading="lazy">` 或 IntersectionObserver | 视口外的图片不加载 |
| **CDN + 格式** | WebP/AVIF 格式，CDN 分发 | 体积小 30-50%，加载快 |
| **响应式图片** | `<picture>` + `srcset` 按屏幕尺寸加载不同图 | 手机不加载桌面大图 |
| **占位** | 模糊占位（blur URL）、Skeleton、背景色 | 防止布局偏移（CLS） |
| **预加载** | `<link rel="preload">` 提前加载关键图片 | LCP 图片优先加载 |
| **雪碧图** | 小图标合并成一张图 | 减少 HTTP 请求数 |
| **缓存** | CDN 缓存 + Service Worker | 重复访问秒开 |

**重点**：
- **懒加载**：最基础的手段，所有非首屏图片都应懒加载
- **WebP/AVIF**：next/image 或图片服务自动转换，几乎无成本
- **响应式图片**：用 `srcset` 让手机加载小图、桌面加载大图，省带宽提速度

```ts
// API: /api/district?adcode=310115&simplify=0.001
// 后端从全国库捞 + 几何简化

// 方案二：前端裁剪
function getDistrict(features: Feature[], adcode: string) {
  return features.filter(f => f.properties.adcode === adcode)
}
```

```
第 4 步：配合矢量瓦片（MVT），拖到哪加载哪，不一次性加载全国
```

## 计算机网络
### 页面要同时订阅车辆位置、报警事件、设备状态，怎么复用WS连接且不丢数据?
:::info **核心原则**
一个 WS 连接 + 按消息 type 分发 + 断线缓存 + seq 校验补数据
:::

#### 1. 架构
```
WS连接(单例) → 分发器 → 车辆位置组件(subscribe 'position')
                      → 报警组件(subscribe 'alarm')
                      → 设备状态组件(subscribe 'status')
```

#### 2. WSClient 单例 + 按 type 分发
```ts
type Handler = (data: any) => void

class WSClient {
  private ws: WebSocket | null = null
  private handlers = new Map<string, Set<Handler>>()
  private offlineBuffer: any[] = []   // 断线缓存
  private lastSeq = 0                  // 服务端消息的递增 seq

  connect(token: string) {
    this.ws = new WebSocket(`wss://api.com/ws?token=${token}`)

    this.ws.onmessage = (e) => {
      const msg = JSON.parse(e.data)   // { type, seq, data }
      this.lastSeq = msg.seq
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
    }

    this.ws.onclose = () => {
      setTimeout(() => this.connect(token), 3000)
    }

    this.ws.onopen = () => {
      this.flushBuffer()                // 重连成功 → 消费缓存
      this.fetchMissed()                 // 拉取断线期间遗漏数据
    }
  }

  // 组件调用
  subscribe(type: string, handler: Handler) {
    if (!this.handlers.has(type)) this.handlers.set(type, new Set())
    this.handlers.get(type)!.add(handler)
    this.send({ type: 'subscribe', channels: [type] })
    return () => this.handlers.get(type)?.delete(handler)
  }

  // 重连后：先消费本地缓存，再 HTTP 补拉遗漏
  private async flushBuffer() {
    while (this.offlineBuffer.length) {
      const msg = this.offlineBuffer.shift()!
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
    }
  }
  private async fetchMissed() {
    const res = await fetch(`/api/ws/missed?from=${this.lastSeq + 1}`)
    const list = await res.json()
    list.forEach((msg: any) => {
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
      this.lastSeq = msg.seq
    })
  }
}
```

#### 3. 组件使用
```tsx
function VehicleMap() {
  useEffect(() => wsClient.subscribe('position', setVehicles), [])
}
function AlarmPanel() {
  useEffect(() => wsClient.subscribe('alarm', v => setAlarms(p => [...p, v])), [])
}
function DeviceStatus() {
  useEffect(() => wsClient.subscribe('status', setDevices), [])
}
```
三个组件共享一个 WS 连接，互不干扰。

#### 4. 不丢数据的保证
| 阶段 | 机制 |
|---|---|
| 连接正常 | 消息带递增 `seq`，客户端可检测跳号 |
| 连接断开 | 数据缓存在 `offlineBuffer` |
| 重连成功 | 先消费缓存，再 `HTTP GET /api/ws/missed?from=seq` 补拉遗漏 |
| 组件卸载 | cleanup 取消订阅，不影响其他组件 |

#### 5. 多 Tab 场景：SharedWorker
不需要每个 tab 都建一个 WS：
```ts
const ws = new WebSocket('wss://api.com/ws')
const ports = new Set<MessagePort>()
const cache: any[] = []

ws.onmessage = (e) => {
  const msg = JSON.parse(e.data)
  cache.push(msg)
  ports.forEach(p => p.postMessage(msg))
}

self.onconnect = (e) => {
  const port = e.ports[0]
  ports.add(port)
  cache.forEach(msg => port.postMessage(msg))
  port.onclose = () => ports.delete(port)
}
```

#### 总结
:::success **面试要点**
答这个题的关键是"**知道怎么防丢**"：*单连接 + 按 type 分发解决多连接浪费、seq 递增检测跳号、断线缓存 + HTTP 补拉保证不丢、SharedWorker 解决多 tab 重复建连。*
:::

### HTTP请求方法有哪些? 各自什么场景?
#### [8种方法](/docs/来时路/计算机网络八股/#常见的http请求方法)
| 方法 | 含义 | 典型场景 |
|---|---|---|
| GET | 获取资源 | 查询列表、获取详情 |
| POST | 提交新资源 | 创建订单、提交表单 |
| PUT | 全量更新 | 更新用户全部信息 |
| PATCH | 部分更新 | 只修改用户头像 |
| DELETE | 删除资源 | 删除一条记录 |
| HEAD | 只获取响应头 | 检查资源是否存在、获取文件大小 |
| OPTIONS | 查询支持的请求方法 | 跨域预检请求（CORS preflight） |
| CONNECT | 建立隧道 | HTTPS 代理 |

#### 幂等性
| 方法 | 幂等 | 说明 |
|---|---|---|
| GET / HEAD / OPTIONS | ✅ | 读操作，多次请求结果不变 |
| PUT / DELETE | ✅ | 多次更新/删除同一个资源，结果一致 |
| POST | ❌ | 每次创建新资源，多次调用会创建多条 |
| PATCH | ❌ | 部分更新可能依赖当前状态 |

#### RESTful规范
增 POST → 删 DELETE → 改 PUT/PATCH → 查 GET

### [同源策略](/docs/来时路/浏览器原理八股#同源策略)是什么? 怎么解决[跨域](/docs/计算机网络/跨域)?
**定义**：浏览器的安全机制，限制不同源（协议、域名、端口任一不同）的文档或脚本互相访问。

#### 限制的内容
- ❌ 跨域 DOM 访问（`iframe` 无法操作不同源的页面内容）
- ❌ 跨域 AJAX 请求（`XMLHttpRequest` / `fetch` 默认被拦截）
- ✅ 跨域资源引用不受限（`<script>`、`<img>`、`<link>` 可以加载）

#### 跨域解决方案
| 方案 | 原理 | 适用场景 |
|---|---|---|
| CORS | 服务端返回 `Access-Control-Allow-Origin` 头 | **标准方案，首选** |
| [JSONP](/docs/情景与手撕题/实现jsonp) | 利用 `<script>` 不受同源策略限制 | 仅 GET 请求，兼容老旧浏览器 |
| 反向代理 | 同源路径代理到目标服务器（如 Vite/webpack proxy） | 开发环境，或 Nginx 转发 |
| postMessage | `window.postMessage` 跨文档通信 | iframe 通信、跨域窗口通信 |
| WebSocket | 协议本身支持跨域 | 实时通信 |

#### CORS响应头字段
| 响应头 | 说明 | 示例 |
|---|---|---|
| `Access-Control-Allow-Origin` | 允许的源 | `*` 或 `https://example.com` |
| `Access-Control-Allow-Methods` | 允许的请求方法 | `GET, POST, PUT, DELETE` |
| `Access-Control-Allow-Headers` | 允许的自定义头 | `Authorization, Content-Type` |
| `Access-Control-Allow-Credentials` | 是否允许带 cookie | `true`（不能和 `*` 同时用） |
| `Access-Control-Max-Age` | 预检结果缓存时间（秒） | `86400` |

**简单请求 vs 非简单请求**：
- **简单请求**：GET/POST/HEAD + 无自定义头 + Content-Type 限 `text/plain`/`multipart`/`x-www-form-urlencoded` → 直接发，不触发预检
- **非简单请求**：PUT/DELETE/PATCH + 自定义头（如 `Authorization`）+ `Content-Type: application/json` → **先发 OPTIONS 预检**，通过后才发真实请求

#### [JSONP原理](/docs/情景与手撕题/实现jsonp)
动态创建 `<script>` 标签，请求 URL 带 `?callback=fn`，服务端返回 `fn({data})`，浏览器作为 JS 直接执行拿到数据。仅支持 GET。

**可跨域加载资源的标签**：`<script>`、`<img>`、`<link>`（仅 CSS）、`<video>`/`<audio>`、`<iframe>`。浏览器允许加载但**不允许 JS 读取内容**（除非 CORS）。

#### 代理服务器跨域原理
跨域限制是浏览器的安全策略，服务端之间没有跨域限制。浏览器请求同域代理地址，代理转发到目标服务器再返回结果——浏览器全程只跟自己的域通信，不触发同源策略。

### 点击图表某个项目，地图跟着联动展示，怎么处理?
:::info **核心思路**
统一状态管理 + 事件总线 + 数据过滤
:::

#### 1. 整体架构
```
图表点击 → 更新全局选中状态 → 地图监听状态变化 → 地图高亮/聚焦
                    ↓
              其他面板（详情、列表等）同步更新
```

#### 2. 统一状态管理
不要在各组件内部各自维护状态，而是抽到上层统一管理：
```ts
// 全局选中状态
interface GlobalSelection {
  selectedId: string | null
  selectedType: "city" | "vehicle" | "device" | null
  hoveredId: string | null
  filter: Record<string, any>
}

const selectionStore = {
  state: { selectedId: null, selectedType: null, hoveredId: null, filter: {} },
  listeners: new Set<() => void>(),

  setSelected(id: string, type: string) {
    this.state.selectedId = id
    this.state.selectedType = type
    this.notify()
  },

  setFilter(filter: Record<string, any>) {
    this.state.filter = filter
    this.notify()
  },

  subscribe(cb: () => void) {
    this.listeners.add(cb)
    return () => this.listeners.delete(cb)
  },

  notify() {
    this.listeners.forEach(cb => cb())
  },
}
```

#### 3. 图表组件：点击时发布
```tsx
function CityChart({ data }: { data: City[] }) {
  const handleClick = (city: City) => {
    selectionStore.setSelected(city.id, "city")
    selectionStore.setFilter({ region: city.region })
  }
  return (
    <BarChart data={data}>
      <Bar dataKey="value" onClick={handleClick} />
    </BarChart>
  )
}
```

#### 4. 地图组件：订阅变化后更新
```tsx
function MapView() {
  const mapRef = useRef<Map>()
  const [filter, setFilter] = useState({})

  useEffect(() => {
    return selectionStore.subscribe(() => {
      const { selectedId, selectedType, filter } = selectionStore.state
      setFilter(filter)
      if (selectedId && mapRef.current) {
        mapRef.current.flyTo({ center: getCenter(selectedId), zoom: 12 })
        mapRef.current.setFeatureState(
          { source: "boundaries", id: selectedId },
          { selected: true }
        )
      }
    })
  }, [])
  return <MapContainer ref={mapRef} />
}
```

#### 5. 数据过滤策略
| 操作 | 数据变化 | 地图响应 |
|---|---|---|
| 点击省份 | `filter.region = province` | 地图飞入该省 |
| 点击城市 | `filter.cityId = cityId` | 聚焦城市，高亮边界 |
| 点击设备 | `filter.deviceId = deviceId` | 定位坐标，弹窗详情 |
| 取消选中 | `filter = {}` | 回到全览 |

#### 6. 跨组件 hover 联动
```tsx
<Bar dataKey="value"
  onMouseEnter={(_, index) => selectionStore.setHovered(data[index].id)}
  onMouseLeave={() => selectionStore.setHovered(null)}
/>
```
地图监听 `hoveredId` 变化，对应标记加光圈/变色。

#### 7. 防抖
```ts
useEffect(() => {
  const timer = setTimeout(() => {
    mapRef.current?.flyTo({ center: getCenter(selectedId), zoom: 12 })

```

图表和地图联动本质是"**发布-订阅**"：*统一状态层管理选中/过滤状态、图表发布点击事件、地图订阅后 flyTo + 高亮、hover 跨组件同步、动画加防抖。*
:::

### WebSocket和Socket.IO有什么区别?
| | WebSocket | Socket.IO |
|---|---|---|
| 本质 | 通信协议（标准） | 封装库（WebSocket + 降级方案） |
| 兼容性 | 现代浏览器 | 自动降级 HTTP 长轮询 |
| 重连 | 需手动实现 | 内置自动重连 |
| 心跳 | 需手动实现 | 内置 ping/pong |
| 事件 | onmessage | emit/on 命名事件 |
| 房间 | 不支持 | 支持房间/命名空间 |

**总结**：WebSocket 是底层协议，Socket.IO 封装了重连/心跳/事件/房间，开箱即用。

### SSE 和 WebSocket 的区别?
| | SSE（Server-Sent Events） | WebSocket |
|---|---|---|
| 方向 | 服务端→客户端（**单向**） | **双向**（全双工） |
| 协议 | HTTP（基于 EventSource API） | WS 协议（独立握手） |
| 数据格式 | 纯文本（自动 UTF-8） | 文本或二进制 |
| 自动重连 | **内置**（断线自动重连） | 需手动实现 |
| 浏览器支持 | 除 IE 外都支持 | 现代浏览器都支持 |
| 跨域 | 支持（通过 `withCredentials`） | 协议本身不受同源限制 |

### 应用场景
- 只需要服务端推送消息（通知、股价、日志）→ **SSE**，简单轻量，自带重连
- 需要双向通信（聊天、游戏、协同编辑）→ **WebSocket**

#### 如何发起 SSE 请求
浏览器原生 `EventSource` API：
```js
const source = new EventSource('/api/notifications')

// 监听命名事件
source.addEventListener('new-order', (e) => {
  console.log('新订单:', JSON.parse(e.data))
})

// 监听未命名的消息
source.onmessage = (e) => { console.log(e.data) }

// 监听错误（断线会自动重连）
source.onerror = (e) => { console.log('连接异常，浏览器会自动重连') }

// 关闭连接
source.close()
```
**关键点**：`EventSource` 只支持 GET 请求，默认自带断线重连，服务端返回格式为 `data: {...}\n\n`。

#### fetch 和 EventSource 的区别
| | fetch | EventSource |
|---|---|---|
| 方向 | **请求-响应**（一问一答） | **服务端推送**（持续流） |
| 连接 | 短连接，请求完即断 | **长连接**，保持打开 |
| 重连 | 需手动处理 | **自动重连** |
| 方法 | 支持 GET/POST/PUT/DELETE | 只支持 GET |
| 数据格式 | 任意（JSON/文本/二进制） | 纯文本（`data: ...` 格式） |
| 适用场景 | 普通 API 调用、增删改查 | 实时通知、推送、日志流 |

- **什么时候用 fetch**：一次性的请求-响应，如获取用户列表、提交表单。
- **什么时候用 EventSource**：服务端持续推送数据，如实时通知、股价更新、日志流。

#### 总结
SSE 是服务端单向推送，开箱即用带重连；WebSocket 是双向通道，灵活但自己处理重连。

## 文件处理
### 大文件分片上传的原理，如何分片，断点续传，如何记录上传进度?
:::info
**核心流程**：文件切片 → 并发上传 → 服务端合并 → 断点续传依赖唯一标识
:::

#### 1. 整体流程
```
文件选择 → 计算 hash → 切分成片 → 并发上传 → 服务端合并
                    ↓                       ↑
              记录已上传的片             断点续传(跳过已上传)
```

#### 2. 文件分片
```ts
const CHUNK_SIZE = 5 * 1024 * 1024  // 每片 5MB

function createChunks(file: File): Blob[] {
  const chunks: Blob[] = []
  let start = 0
  while (start < file.size) {
    const end = Math.min(start + CHUNK_SIZE, file.size)
    chunks.push(file.slice(start, end))
    start = end
  }
  return chunks
}
```

#### 3. 计算文件 hash（唯一标识）
用 `spark-md5` 在浏览器端计算文件 hash，用于断点续传时服务端识别同一文件：
```ts
import SparkMD5 from 'spark-md5'

function calcFileHash(file: File): Promise<string> {
  return new Promise((resolve, reject) => {
    const spark = new SparkMD5.ArrayBuffer()
    const reader = new FileReader()
    const chunks = createChunks(file)

    let i = 0
    reader.onload = (e) => {
      spark.append(e.target!.result as ArrayBuffer)
      i++
      if (i < chunks.length) {
        reader.readAsArrayBuffer(chunks[i])
      } else {
        resolve(spark.end())  // 得到文件唯一 hash
      }
    }
    reader.onerror = reject
    reader.readAsArrayBuffer(chunks[0])
  })
}
```

#### 4. 上传 + 进度记录
```ts
async function uploadFile(file: File) {
  const fileHash = await calcFileHash(file)
  const chunks = createChunks(file)
  const total = chunks.length
  let uploaded = 0

  // 先问服务端：哪些片已经上传过（断点续传）
  const { uploadedIndexes } = await fetch(`/api/upload/status?hash=${fileHash}`).then(r => r.json())
  const uploadedSet = new Set(uploadedIndexes)

  // 并发上传未上传的分片（限制并发数）
  const concurrency = 3
  const tasks = chunks.map((chunk, index) => async () => {
    if (uploadedSet.has(index)) {
      uploaded++  // 跳过已上传的
      updateProgress(uploaded / total)
      return
    }
    const formData = new FormData()
    formData.append('chunk', chunk)
    formData.append('hash', fileHash)
    formData.append('index', String(index))
    formData.append('total', String(total))

    await fetch('/api/upload/chunk', { method: 'POST', body: formData })
    uploaded++
    updateProgress(uploaded / total)
  })

  // 并发控制
  await concurrentLimit(tasks, concurrency)

  // 通知服务端合并
  await fetch('/api/upload/merge', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ hash: fileHash, total }),
  })
}

function updateProgress(percent: number) {
  // 更新进度条
  progressBar.style.width = `${(percent * 100).toFixed(1)}%`
}
```

#### 5. 断点续传的关键
```
上传中断 → 重新选择同一文件（hash 相同）
         → 请求 /api/upload/status?hash=xxx
         → 服务端返回已收到的分片索引 [0, 1, 3, 5, ...]
         → 客户端只上传缺失的分片 [2, 4, 6, ...]
         → 合并时校验完整性
```

#### 6. 服务端伪代码
```ts
// /api/upload/status — 查询哪些片已上传
app.get('/api/upload/status', (req, res) => {
  const { hash } = req.query
  const uploaded = redis.smembers(`upload:${hash}`)  // 存 Set 结构
  res.json({ uploadedIndexes: uploaded.map(Number) })
})

// /api/upload/chunk — 接收单片
app.post('/api/upload/chunk', upload.single('chunk'), (req, res) => {
  const { hash, index } = req.body
  const chunkPath = path.join(UPLOAD_DIR, hash, index)
  fs.mkdirSync(path.dirname(chunkPath), { recursive: true })
  fs.renameSync(req.file.path, chunkPath)
  redis.sadd(`upload:${hash}`, index)
  res.json({ ok: true })
})

// /api/upload/merge — 合并所有分片
app.post('/api/upload/merge', (req, res) => {
  const { hash, total } = req.body
  const writeStream = fs.createWriteStream(path.join(UPLOAD_DIR, `${hash}.zip`))
  for (let i = 0; i < total; i++) {
    writeStream.write(fs.readFileSync(path.join(UPLOAD_DIR, hash, i)))
  }
  writeStream.end()
  res.json({ url: `/files/${hash}.zip` })
})
```

#### 总结
:::success **面试要点**
分片上传的核心是*前端 hash 去重 + 分片并发 + 服务端按索引合并*。断点续传依赖 hash 标识已上传的片，跳过重传。进度 = 已上传片数 / 总片数。
:::

## React
### [useEffect和useLayoutEffect的区别](/docs/来时路/React八股#useeffect与uselayouteffect的区别)
:::info **核心区别**
执行时机不同
:::

| | `useEffect` | `useLayoutEffect` |
|---|---|---|
| 执行时机 | 浏览器绘制**之后**异步执行 | 浏览器绘制**之前**同步执行 |
| 阻塞绘制 | 不阻塞 | 会阻塞 |
| 适用场景 | 数据请求、日志、订阅 | DOM 测量、动画、需同步读布局 |

**执行顺序**：组件渲染 → DOM 更新 → `useLayoutEffect`（绘制前） → 浏览器绘制 → `useEffect`（绘制后）

**怎么选**：不涉及 DOM 测量就用 `useEffect`（默认）；需要在绘制前拿到 DOM 尺寸或执行动画就用 `useLayoutEffect`，否则用户会看到闪烁。

### [Vue和React的diff算法区别](/docs/来时路/React八股#react与vue的diff算法区别)
**相同点**：都是基于虚拟DOM的同层比较，不做跨层级复用。

**关键区别**：
| | Vue | React |
|---|---|---|
| 对比策略 | 双端指针（头尾往中间比） | 从左到右单次遍历 |
| key 的作用 | 判断可复用的节点（移动而非重建） | 判断可复用的 Fiber 节点 |
| 列表更新 | 头尾各两个指针，尽可能复用 | 只从头开始，标记移动位置 |
| 更新粒度 | 组件级 + 响应式追踪（精确到值） | Fiber 架构 + 时间分片（整棵树 diff） |

**本质差异**：
- **Vue**有响应式系统，知道哪个数据变了、变了多少，diff时直接锁定变化范围，双端指针尽可能复用旧节点，减少移动
- **React**没有响应式追踪，不知道数据哪变了，只能整棵树重diff，靠Fiber分片执行、可中断

**一句话**：Vue的diff是"知道哪变了再去比"（响应式驱动）；React的diff是"不知道哪变了所以全比一遍"（无侵入式）。
| 维度 | Vue | React |
|---|---|---|
| 范式 | 声明式 + 响应式 | 函数式 + 不可变 |
| 数据流 | 双向绑定（v-model） | 单向数据流 |
| 状态管理 | data 直接修改（响应式代理） | setState(不可变更新) |
| 模板 | 模板语法 + 指令（v-if/v-for） | JSX（JavaScript in HTML） |
| diff 策略 | 双端指针 + 静态标记 | Fiber + 单端遍历 |
| 更新粒度 | 响应式追踪，精确到组件 | 顶往下全量 diff，Fiber 可中断 |
| 上手难度 | 低（模板直观） | 中（JSX + 函数式概念） |
| 适用场景 | 后台管理、快速交付 | 大型复杂应用、需精细控制 |

**核心差异**：
- **响应式**：Vue 用 Proxy 代理数据，修改 data 自动触发更新；React 需要手动 setState，用不可变数据触发重新渲染
- **模板 vs JSX**：Vue 模板接近 HTML，有 v-if/v-for 指令；React 全 JS，`if/for` 就是 JS 本身的语法
- **更新机制**：Vue 知道哪个数据变了，精确更新对应组件；React 不知道哪变了，从根组件开始整棵树 diff，靠 Fiber 分片不卡 UI

### React Diff 算法的时间复杂度
React diff 的平均时间复杂度是 **O(n)**。传统的树 diff 是 O(n³)，React 通过三个策略压到 O(n)：
| 策略 | 做法 | 效果 |
|---|---|---|
| 同层比较 | 只对比同一层级，不跨层级 | O(n³) → O(n²) |
| 类型判断 | 节点类型不同直接销毁重建 | 大幅减少对比 |
| key 属性 | 列表用 key 快速匹配新旧节点 | 列表 diff O(n²) → O(n) |

**总结**：同层比、类型不同直接换、有 key 找 key。

### useState 为什么返回数组而不是对象?
**为了解构时灵活命名**。数组解构时变量名由调用者决定：
```js
const [count, setCount] = useState(0)
const [name, setName] = useState('')
const [items, setItems] = useState([])
```

如果返回对象 `{ state, setState }`，解构时必须重命名：
```js
const { state: count, setState: setCount } = useState(0)
```
每次用都要 `{ state: xxx, setState: setXxx }`，繁琐且容易冲突。数组解构直接随意取名，简洁很多。

**总结**：数组解构让调用方自己取名，比对象解构重命名简洁得多。这是 React 团队刻意的 API 设计选择。

### 为什么 Hooks 必须在顶层调用?
**因为 React 依赖调用顺序来匹配 Hook 状态**。React 内部用链表记录每个 Hook 的值，第一次渲染时按调用顺序依次注册：
```
第一次渲染:  useState(0)  → 链表节点1  (count)
             useState('') → 链表节点2  (name)
             useEffect()  → 链表节点3  (effect)
```
后续渲染 React **按顺序逐个读取**这些节点。如果放在条件或循环中：
```js
if (condition) {
  useState(0)  // 条件不满足时跳过了 → 链表错乱
}
useState('')   // 本该匹配节点2，但上一轮的节点2是 useState(0)
```
跳过一个 Hook 后，后面的所有 Hook 的索引全部对不上，导致状态错乱或报错。

**总结**：React 用**调用顺序**标识每个 Hook，放在条件/循环/嵌套里会被打破顺序，导致状态错乱。

### setState 内部做了哪些事?
#### setState的执行流程
```
setState(newValue)
  → 1. 合并新状态到队列（不是立即修改）
  → 2. 调度更新（标记该组件需要重新渲染）
  → 3. 进入协调阶段（Reconciliation）
     → 3.1 执行组件函数，得到新的虚拟 DOM
     → 3.2 新旧虚拟 DOM 进行 diff 对比
     → 3.3 计算需要更新的 DOM 操作（最小化变更）
  → 4. 提交阶段（Commit）
     → 4.1 将变更应用到真实 DOM
     → 4.2 执行 useLayoutEffect
     → 4.3 浏览器绘制
     → 4.4 执行 useEffect
```

#### 关键特性
| 特性 | 说明 |
|---|---|
| **异步更新** | 连续调用 `setCount(n+1)` 多次只会合并成一次重新渲染 |
| **批量处理** | 同一事件循环内的 `setState` 会批量合并，不重复渲染 |
| **函数式更新** | `setCount(prev => prev + 1)` 能拿到最新值解决闭包陷阱 |
| **类组件 vs 函数组件** | 类组件 `setState` 是合并对象；函数组件 `setXxx` 是直接替换

### Zustand 为什么只适合轻量项目?
Zustand 的核心设计是**轻量 + 无模板**，但它的优缺点决定了适用场景：
| | Zustand | Redux Toolkit | MobX |
|---|---|---|---|
| 学习成本 | 低 | 中 | 中 |
| 模板代码 | 极少 | 中 | 少 |
| 中间件生态 | 有限 | 丰富 | 中 |
| 大规模状态 | 不适合（无规范化约束） | 适合（slice + entity） | 适合 |
| DevTools | 基础 | 完善 | 完善 |

#### 限制原因
- **无状态规范化**：没有类似 Redux Toolkit 的 `createEntityAdapter`，多个 Store 间数据冗余难维护
- **中间件有限**：复杂场景（如撤销重做、持久化逻辑）缺少现成中间件
- **单 Store 模式**：默认不拆分 Store，数据量大了易膨胀

#### 结论
中后台表单、简单数据流场景 Zustand 很香；多人协作的大项目、复杂状态交互建议 Redux Toolkit。

### Zustand 为什么不需要 Provider? 底层实现原理是什么?
**不需要 Provider** 的核心原因：状态存在**模块作用域**里，而不是存在 React 组件树里。

#### 底层简化实现
```ts
const listeners = new Set()
let state

const store = {
  getState: () => state,
  setState: partial => { state = { ...state, ...partial }; listeners.forEach(l => l()) },
  subscribe: l => listeners.add(l),
}
// useStore 只是订阅 store 变化触发组件重渲染
```

#### 为什么不用 Provider
Redux 的 store 通过 Provider 的 Context 注入组件树；Zustand 的 store 是模块级变量，组件直接 `import` 使用。

#### 组件怎么知道要重新渲染
`useSyncExternalStore` 是 React 18 内置的 Hook，它允许外部 store 接入 React 的更新机制。内部原理：
```
组件渲染调用 useSyncExternalStore(subscribe, getSnapshot)
  → React 在当前 fiber 上注册该外部 store
  → 调用 subscribe(callback)，其中 callback 是 React 内部的 forceStoreRerender
  → 外部触发 setState → 遍历 listeners → 执行 callback(forceStoreRerender)
  → React 收到 callback 后将该 fiber 标记为"需要更新"（和 setState 走同一套调度）
  → 重新渲染组件，调用 getSnapshot() 拿到新值
  → 新旧快照对比（Object.is），没变则跳过渲染
```
**核心机制**：`useSyncExternalStore` 的 subscribe callback 底层调用了 React 的`请求更新`（和 `setState` 走同一个调度队列），所以不管调用方是 React 内部还是外部，都能触发组件重渲染。

### React 18新增了哪些Hooks
| Hook | 作用 |
|---|---|
| `useId` | 生成唯一 ID，解决 SSR 时客户端服务端 ID 不一致 |
| `useTransition` | 标记低优先级更新，不阻塞 UI（如搜索建议） |
| `useDeferredValue` | 延迟更新某个值，类似防抖但由 React 调度 |
| `useSyncExternalStore` | 让外部 store（Zustand/Redux）接入 React 的并发更新 |
| `useInsertionEffect` | 在 DOM 变更前同步执行，专用于 CSS-in-JS 注入样式 |

**重点**：
- `useTransition`：大列表搜索时，用户输入保持流畅，结果渲染延迟
- `useSyncExternalStore`：解决外部 store 在并发模式下读取不一致的问题
- `useId`：解决 SSR 场景下服务端和客户端生成的 ID 不匹配

### 对纯函数的理解
**两个条件缺一不可**：相同输入始终相同输出 + 没有副作用。
```js
const add = (a, b) => a + b  // ✅ 纯函数
let count = 0; const inc = () => count++  // ❌ 副作用
```
**为什么前端关注纯函数**：React 的 `useReducer`、Redux 的 reducer 都要求纯函数——保证状态更新可预测、可回溯、可测试。

## Next.js
### Next.js 有哪些渲染方式?
| 方式 | 全称 | 渲染时机 | 数据源 | 适用场景 |
|---|---|---|---|---|
| **SSR** | 服务端渲染 | 每次请求时 | 实时数据 | 动态内容、SEO 重要 |
| **SSG** | 静态生成 | 构建时 | 编译时确定 | 博客、文档、营销页 |
| **ISR** | 增量静态再生成 | 构建时+定时 | revalidate 间隔 | 内容更新的静态站 |
| **CSR** | 客户端渲染 | 浏览器端 | API 请求 | 后台管理、仪表盘 |

**如何选**：内容固定→SSG；动态需SEO→SSR；需SSG速度但会更新→ISR；纯交互→CSR。

### 怎么做[SEO](/docs/Web开发与安全/SEO)?
| 方向 | 具体手段 |
|---|---|
| **服务端渲染** | SSG/SSR 确保爬虫能拿到完整的 HTML（CSR 爬不到 JS 渲染的内容） |
| **元数据** | `<title>` + `<meta name="description">` + `<meta name="keywords">`，每页独立 |
| **语义化标签** | `<header>` `<nav>` `<main>` `<article>` `<aside>` `<footer>`，用 `<h1>`~`<h6>` 体现层级 |
| **结构化数据** | 嵌入 JSON-LD（Schema.org），让搜索引擎理解内容类型（文章/产品/FAQ） |
| **图片优化** | `alt` 属性描述图片内容，`loading="lazy"` 延迟加载 |
| **链接建设** | 清晰的站内导航、面包屑导航、内链结构 |
| **性能** | LCP（加载速度）、CLS（布局稳定性）、FID（交互响应），Core Web Vitals 是排名因素 |
| **Sitemap** | 提交 `sitemap.xml` 给搜索引擎，`robots.txt` 控制爬虫权限 |

**总结**：让爬虫能读到（SSR/SSG）、能读懂（语义标签 + 结构化数据）、读得快（性能优化）。

### 文件夹即路由的实现原理
Next.js App Router / Nuxt.js 等框架的核心机制：**文件系统映射到 URL 路径**。

#### 原理
```
项目目录                          URL 路径
pages/
  index.tsx      →  /             (首页)
  about.tsx      →  /about
  blog/
    index.tsx    →  /blog
    [id].tsx     →  /blog/123     (动态路由)
  api/
    hello.ts     →  /api/hello    (API 路由)
```

#### 底层实现（简化思路）
```ts
// 构建时扫描文件系统，生成路由表
const routeMap = {
  '/':              { file: 'pages/index.tsx',     params: [] },
  '/about':         { file: 'pages/about.tsx',    params: [] },
  '/blog':          { file: 'pages/blog/index.tsx', params: [] },
  '/blog/:id':      { file: 'pages/blog/[id].tsx', params: ['id'] },
  '/api/hello':     { file: 'pages/api/hello.ts',  params: [] },
}

// 请求时，根据 URL 匹配路由表
function matchRoute(url) {
  for (const [pattern, route] of Object.entries(routeMap)) {
    const regex = pathToRegex(pattern) // /blog/:id → /blog/([^/]+)
    const match = url.match(regex)
    if (match) return { route, params: extractParams(match) }
  }
  return null // 404
}
```

#### 总结
构建工具（Webpack/Vite）在编译时扫描 `pages/` 目录，按文件路径生成路由配置表，运行时根据 URL 匹配对应组件并渲染。`[param]` 文件名的方括号语法用来标识动态参数。

### Next.js 中的水合（Hydration）和同构（Isomorphic）是什么
#### 同构（Isomorphic）
同一套代码在服务端和客户端都运行。Next.js的组件先在服务端渲染成HTML发给浏览器，然后在浏览器端"激活"为可交互的React组件。
#### 水合（Hydration）
```
服务端渲染 HTML（静态，不可交互）
  ↓ 浏览器收到 HTML，立刻展示内容
  ↓ 加载 JS bundle
  ↓ React 在浏览器端"水合"——复用已有的 DOM，绑定事件处理器
  ↓ 页面变为可交互
```

#### 水合的过程
1. 服务端将组件渲染为 HTML 字符串，内嵌到页面中返回
2. 浏览器直接展示 HTML（首屏内容可见）
3. 浏览器下载并执行 JS
4. React 在内存中重建虚拟 DOM
5. React 将虚拟 DOM 与服务端已有的真实 DOM 对比（差异很小，因为同构）
6. React 绑定事件、state 等 JavaScript 能力，DOM 节点"激活"

#### 服务端数据如何给到客户端?
服务端在 HTML 中嵌入 `<script id="__NEXT_DATA__">` 标签，把数据序列化为 JSON。浏览器端水合时直接读这个标签拿到数据。
```
getServerSideProps → 数据传给组件渲染 HTML
                  → 同样数据写入 __NEXT_DATA__ 嵌入 HTML
浏览器端 → 展示 HTML → React 从 __NEXT_DATA__ 读取数据 → 水合
```
这样就保证了服务端和客户端渲染数据一致，不会闪。

#### 总结
服务端渲染 HTML 让首屏快，浏览器水合同构的 React 组件让页面可交互，两者结合就是 Next.js SSR 的核心。

### [闭包](/docs/来时路/JavaScript八股#对闭包的理解)的使用场景
**定义**：函数 + 它引用的外部变量，即使外部函数执行完，内部函数还能访问那些变量。

#### 常用场景
| 场景 | 原理 | 示例 |
|---|---|---|
| 数据私有化 | 变量不暴露到全局 | 计数器、缓存 |
| [函数柯里化](/docs/情景与手撕题/柯里化currying的实现) | 固定部分参数，返回新函数 | `const add = x => y => x + y` |
| [防抖/节流](/docs/Web开发与安全/防抖与节流) | 保存 timer 引用 | 搜索输入、滚动事件 |
| [高阶组件（React）](/docs/来时路/React八股#高阶组件hoc) | 记住外部传入的配置 | HOC 模式 |
| 循环中的异步 | 用 IIFE 保存每次循环的 i | `for` 中 `setTimeout` |

#### 闭包陷阱
循环中用闭包取到的变量是最终值而非当前值，需要用IIFE（立即调用函数表达式，immediately invoked function expression）或`let`解决。

#### 总结
**一句话**：闭包 = 函数 + 它"记住"的外部变量，用于封装私有数据、函数工厂、防抖节流。

### `JSON.parse()`有什么缺点?
| 缺点 | 说明 |
|---|---|
| **不能序列化函数** | 对象中的方法、undefined、Symbol 会被丢弃 |
| **不支持循环引用** | `JSON.parse(JSON.stringify(obj))` 遇到循环引用会抛错 |
| **精度丢失** | 大整数、Date 会被转成字符串再解析，类型不对 |
| **安全问题** | `JSON.parse('{"__proto__": {"x": 1}}')` 可能污染原型链（旧引擎） |
| **性能** | 大 JSON 字符串解析耗时长，会阻塞主线程 |

#### 替代方案
- 需要序列化函数/循环引用 → 用 `structuredClone()`（深拷贝）或自定义序列化
- 大 JSON 解析不阻塞主线程 → 用 `JSON.parse()` 在 Web Worker 中执行

#### 总结
JSON.parse 只能处理纯数据（字符串/数字/布尔/普通对象/数组），碰到函数、Date、undefined、循环引用就挂了。

### let、const、var 的区别?
| | var | let | const |
|---|---|---|---|
| 作用域 | 函数级 | **块级**（`{}` 内） | 块级 |
| 变量提升 | ✅（值为 undefined） | ✅（但暂时性死区，不能访问） | ✅（但暂时性死区） |
| 重复声明 | ✅ 允许 | ❌ 不允许 | ❌ 不允许 |
| 重新赋值 | ✅ | ✅ | ❌（引用类型可改属性） |
| 挂载到 window | ✅ | ❌ | ❌ |

#### 暂时性死区（TDZ）
`let/const` 声明前访问变量会报 `ReferenceError`，不像 `var` 返回 `undefined`。
```js
console.log(a) // undefined（var 提升）
console.log(b) // ReferenceError（let 暂时性死区）
var a = 1
let b = 2
```

#### const不可变指什么
`const` 保证的是**引用不可变**，不是值不可变。`const obj = {}; obj.x = 1` 可以，`obj = {}` 不行。

#### 最佳实践
默认用 `const`，需要重新赋值用 `let`，永远不用 `var`。

### apply、call、bind 的区别?
**共同点**：都用于改变函数执行时的 `this` 指向。
| | apply | call | bind |
|---|---|---|---|
| 传参方式 | 数组 `[arg1, arg2]` | 逐个传入 `arg1, arg2` | 逐个传入 |
| 执行时机 | **立即执行** | **立即执行** | 返回新函数，**不立即执行** |
| 返回值 | 函数执行结果 | 函数执行结果 | 新函数（可稍后调用） |

#### 用法示例
```js
fn.apply(ctx, [1, 2])     // this = ctx，参数数组
fn.call(ctx, 1, 2)        // this = ctx，参数逐个传
const bound = fn.bind(ctx) // 返回新函数，this 永久绑定为 ctx
bound(1, 2)               // 调用时才执行
```

#### bind的特殊用途
bind 不仅可以绑定 `this`，还能预设部分参数（偏函数）：
```js
const add = (a, b) => a + b
const add5 = add.bind(null, 5)  // 预设 a = 5
add5(3)  // 8
```

#### 总结
apply/call 立即执行，bind 返回新函数稍后执行。传数组用 apply，传列表用 call，想留着以后用或预设参数用 bind。

### 浏览器的sort方法是否稳定?
- **ES2019 之前**：不同浏览器实现不同，Chrome 用 TimSort（稳定），旧 Safari 用其他算法（不稳定）。
- **ES2019 之后**：规范要求 `Array.prototype.sort` 必须是**稳定排序**。所有现代浏览器都遵循。

**什么是稳定排序**：排序时如果两个元素相等，它们原来的相对顺序不变。
```js
// 稳定排序 → {id:1, score:80} 一定在 {id:3, score:80} 前面
students.sort((a, b) => a.score - b.score)
// 如果分数相同，id 1 依然在 id 3 前面
```
**总结**：ES2019 之后 `sort` 是稳定的，所有现代浏览器都保证稳定。实现都是 TimSort（归并+插入的混合排序）。

### [事件循环（Event Loop）](/docs/来时路/浏览器原理八股#事件循环event-loop)是什么?
JS 是单线程的，事件循环是用来协调同步代码、异步回调执行顺序的机制。
#### 执行顺序
```
同步代码 → 微任务（microtask）→ 宏任务（macrotask）→ 渲染
```
#### 任务分类
| 类型 | 事件 | 优先级 |
|---|---|---|
| **同步** | 直接执行的语句 | 最高 |
| **微任务** | Promise.then、MutationObserver、queueMicrotask | 次高 |
| **宏任务** | setTimeout、setInterval、I/O、UI 事件 | 最低 |

#### 执行规则
1. 执行当前宏任务中的同步代码
2. 清空所有微任务队列
3. 取出下一个宏任务执行
4. 回到第 1 步

```js
console.log(1)                    // 同步
setTimeout(() => console.log(2))   // 宏任务
Promise.resolve().then(() => console.log(3))  // 微任务
console.log(4)                    // 同步
// 输出：1, 4, 3, 2
```

#### 面试常问
微任务在每次宏任务之后都会清空，所以 `Promise.resolve().then` 永远比 `setTimeout` 先执行。

### JS 数组去重怎么做?
| 方式 | 写法 | 适用场景 |
|---|---|---|
| **Set** | `[...new Set(arr)]` | **首选**，基础类型 |
| **filter + indexOf** | `arr.filter((v,i)=>arr.indexOf(v)===i)` | 兼容旧浏览器 |
| **Map** | `[...new Map(arr.map(v=>[v,v])).keys()]` | 对象数组按 key 去重 |

**对象数组（Array of Objects）**：数组里每个元素是对象，如 `[{id:1,name:'a'}, {id:2,name:'b'}]`。Set 去重对对象无效，因为每个对象引用不同，`{id:1}` `!==` `{id:1}`。所以对象数组需要用 Map 以某个 key（如 `id`）为标识去重。

**总结**：基础类型（数字/字符串）用 Set；对象数组去重用 Map 以 key 为标识。|

### 如何提取一个函数的参数类型
使用 `Parameters<T>` 工具类型：
```ts
function fetchUser(id: number, name: string) {}
type P = Parameters<typeof fetchUser>  // [id: number, name: string]
type First = P[0]  // number
```

**相关**：`ReturnType<T>` 提取返回值，`Awaited<T>` 提取 Promise 内部类型。

#### 手写实现
```ts
type MyParameters<T extends (...args: any[]) => any> =
  T extends (...args: infer P) => any ? P : never

// 使用
type P = MyParameters<(id: number, name: string) => void>
// [id: number, name: string]
```
**原理**：`infer P` 在条件类型中提取 `...args` 的类型，自动得到元组。

### 模拟接口 mock 能模拟 404 或 500 吗?

### 更新代码后CDN还是命中缓存怎么办?
#### 核心方案：文件名哈希（最推荐）
构建时给文件加 content hash，内容变了文件名就变，CDN 视为新资源，不会命中旧缓存：
```js
// webpack 输出：main.8f3k2d.js（内容变了 hash 就变）
output: { filename: '[name].[contenthash].js' }
```

用户永远请求新文件，旧文件 CDN 过期自动淘汰。

#### 其他方案
| 方案 | 做法 | 缺点 |
|---|---|---|
| **文件名哈希** | `app.a1b2c3.js` | 需构建工具支持，**最推荐** |
| **CDN 刷新** | 手动在云厂商控制台刷新缓存 | 手动操作，容易忘 |
| **版本号参数** | `app.js?v=2` | 代理/CDN 不一定认 query，可能不刷新 |
| **短缓存时间** | `Cache-Control: max-age=600` | 失去 CDN 加速意义 |

#### 总结
用 content hash 做文件名，内容变文件名就变，CDN 自然不命中旧缓存。不需要手动刷新、不需要版本号、用户无感知。

## 浏览器原理
### [浏览器缓存机制](/docs/来时路/浏览器原理八股#浏览器缓存机制)是什么? [强缓存和协商缓存](/docs/来时路/浏览器原理八股#强缓存与协商缓存)有哪些?
#### 强缓存
**强缓存**：不请求服务端，直接读本地缓存。
| 响应头 | 说明 |
|---|---|
| `Expires` | 绝对过期时间（HTTP/1.0，依赖客户端时间） |
| `Cache-Control: max-age=3600` | 相对过期秒数（HTTP/1.1，**优先级高**） |

强缓存命中 → 状态码 `200 (from disk/memory cache)`

#### 协商缓存
**协商缓存**：带缓存标识问服务端，服务端判断是否可用。
| 请求头 | 响应头 | 说明 |
|---|---|---|
| `If-Modified-Since` | `Last-Modified` | 文件最后修改时间 |
| `If-None-Match` | `ETag` | 文件唯一指纹（**优先级高**） |

协商缓存命中 → `304 Not Modified`；未命中 → `200` 返回新资源。

#### 总结
- **缓存流程**：请求 → 有强缓存? → 是：直接用本地缓存。否：发请求带 `If-None-Match` → 服务端判断没变 → 304；变了 → 200。
- **刷新行为**：地址栏回车/链接跳转 → 强+协商都生效；F5 → 仅协商生效；Ctrl+F5 → 全不生效。

### 浏览器两个 tab 怎么实时通信? sessionStorage 能实现吗?
#### 跨tab通信方案
| 方案 | 原理 | 能否跨域 |
|---|---|---|
| `localStorage` + `storage` 事件 | 一个 tab 写 localStorage，同源其他 tab 监听 `storage` 事件 | ❌ 同源 |
| `BroadcastChannel` | 显式的发布-订阅频道 | ❌ 同源 |
| `SharedWorker` | 所有 tab 共享一个 Worker 线程 | ❌ 同源 |
| `postMessage` | 通过 window.open 获取引用或 iframe 通信 | ✅ 可跨域 |
| `Service Worker` | 通过消息中转 | ❌ 同源 |
| Cookie + 轮询 | Cookie 可设置 domain 共享，但需轮询 | ✅ |

#### sessionStorage不能实现
sessionStorage 的作用域是**单个 tab/窗口**，每个 tab 都是独立的存储空间。同源不同 tab 的 sessionStorage 互不隔离。

**为什么**：sessionStorage 设计初衷就是"会话级别存储"，关闭 tab 就清除，不同 tab 之间不共享。所以不能用来跨 tab 通信。

#### 推荐方案
同源场景用 `BroadcastChannel`（最简洁）或 `localStorage + storage 事件`（兼容性最好）；跨域场景用 `postMessage`。

### WebSocket 可以不同源吗?
**可以**。WebSocket 协议本身不受同源策略限制。浏览器允许 JS 创建 `new WebSocket('wss://不同域的地址')`，只要服务端支持即可。

**但握手时会带 Origin 头**：
```
GET /ws HTTP/1.1
Origin: https://example.com    ← 浏览器的源
Upgrade: websocket
```

服务端可以根据 `Origin` 判断是否允许该来源的连接，实现类似 CORS 的访问控制。如果拒绝，直接关闭握手即可。这就是 WebSocket 的"服务端校验来源"机制，而不是浏览器的同源策略拦截。|

## 异步编程
### Promise.all 和 Promise.allSettled 的区别? 数组中有错误分别返回什么?
**共同点**：都接收 Promise 数组，并行执行所有任务。
| | Promise.all | Promise.allSettled |
|---|---|---|
| 遇错误 | **一错即弃**，直接 reject | 等**全部完成**，无论对错 |
| 返回值 | 成功值数组 `[v1, v2]` | 对象数组 `{status, value/reason}` |
| 有错误时 | 返回**第一个错误** | 永不 reject，全部返回 |

**例子**：`[resolve(1), reject('错了'), resolve(3)]`
- `Promise.all` → 直接 catch '错了'，值 1 和 3 丢弃
- `Promise.allSettled` → `[{fulfilled,1}, {rejected,'错了'}, {fulfilled,3}]`

**总结**：全部成功才能继续用 `all`，每个都要知道结果用 `allSettled`。

### Promise如何实现串行与并行调用?如何捕获异常?不用Promise.all如何实现并行?
#### 串行
一个个依次执行，前一个结束才启动下一个。
```js
// 串行：用 for 循环 await
async function serial(tasks) {
  const results = []
  for (const task of tasks) {
    results.push(await task()) // 前一个完成后才执行下一个
  }
  return results
}
// 异常捕获：try...catch 包裹循环，任一失败后面的不再执行
```

#### 并行
同时启动，全部跑完再收结果。
```js
// 并行：先启动所有 Promise
async function parallel(tasks) {
  return Promise.all(tasks.map(t => t()))
}
// 异常捕获：Promise.all 一错即弃；Promise.allSettled 全部等完
```

#### 不用Promise.all实现并行
手动收集结果 + 计数器。
```js
function parallelWithoutAll(tasks) {
  return new Promise((resolve, reject) => {
    const results = []
    let done = 0
    tasks.forEach((task, i) => {
      task()
        .then(r => { results[i] = r; done++; if (done === tasks.length) resolve(results) })
        .catch(reject) // 任一失败整体 reject
    })
  })
}
```

#### 异常捕获策略
| 模式 | 串行 | 并行 |
|---|---|---|
| 遇错中断 | try...catch 包裹循环 | Promise.all → catch |
| 遇错继续 | 每个 task 内部 catch，不抛出 | Promise.allSettled 逐个判断 status |

## CSS
### flex属性的含义?设置成`100`代表什么?
**`flex`是三个属性的简写**：`flex-grow flex-shrink flex-basis`。
```css
flex: 1;        /* flex: 1 1 0   (可扩大、可缩小、初始0)  */
flex: auto;     /* flex: 1 1 auto (可扩大、可缩小、初始auto) */
flex: none;     /* flex: 0 0 auto (不扩大、不缩小) */
flex: 100;      /* flex: 1 1 100px */
```

**`flex: 100`的含义**：等价于 `flex-grow: 1` `flex-shrink: 1` `flex-basis: 100px`。
- `flex-basis: 100px` → 项目初始占据 100px
- `flex-grow: 1` → 容器有剩余空间时，项目等比例放大
- `flex-shrink: 1` → 容器空间不足时，项目等比例缩小

#### 面试常问的单值写法
| 值 | 等价于 | 效果 |
|---|---|---|
| `flex: 1` | `1 1 0` | 等分剩余空间（常用） |
| `flex: auto` | `1 1 auto` | 按内容宽度分配 |
| `flex: none` | `0 0 auto` | 固定宽度，不伸缩 |
| `flex: 0` | `0 1 0` | 不放大，可缩小 |
| `flex: 100` | `1 1 100px` | 初始 100px，可伸缩 |

### [`display: none`和`visibility: hidden`的区别](/docs/来时路/CSS八股#displaynone和visibilityhidden的区别)?
| | display: none | visibility: hidden |
|---|---|---|
| 是否占据空间 | **不占据**（元素从文档流移除） | **占据**（元素不可见但占位） |
| 子元素继承 | 子元素全部消失，无法覆盖 | 子元素可设置 `visibility: visible` 显示 |
| 事件触发 | 无法触发任何事件 | 无法触发（元素区域不可点） |
| 动画过渡 | 不支持 transition | 支持透明度过渡 |
| 重排重绘 | 触发**重排**（影响布局） | 只触发**重绘**（不影响布局） |

**总结**：`display: none` 把元素从布局中彻底移除，影响周围元素；`visibility: hidden` 只是让元素看不见，位置还在。测量元素用 `visibility: hidden` 而不是 `display: none`。

### div 文字溢出怎么用...省略表示?
#### 单行文本
三个属性缺一不可：
```css
.single-line {
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
}
```

#### 多行文本
第 N 行省略：
```css
.multi-line {
  display: -webkit-box;
  -webkit-line-clamp: 2;       /* 第 2 行显示省略号 */
  -webkit-box-orient: vertical;
  overflow: hidden;
}
```

#### 总结
单行用 `text-overflow: ellipsis` + `white-space: nowrap`；多行用 `-webkit-line-clamp`。

### 如何做样式隔离?
**样式隔离**：防止组件之间的 CSS 相互影响。比如 A 组件写了 `.btn { color: red }`，B 组件的按钮不会变红。
| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **CSS Modules** | 编译时给类名加 hash | 无运行时开销 | 全局样式需特殊处理 |
| **Scoped（Vue）** | 编译时加 `data-v-xxx` 属性选择器 | 写法简单 | 选择器权重增加 |
| **CSS-in-JS** | 运行时生成唯一类名 | 动态样式方便 | 运行时开销 |
| **Shadow DOM** | 浏览器原生隔离 | 真正物理隔离 | 兼容性、调试困难 |

**总结**：CSS Modules 最轻量，Vue Scoped 体验好，CSS-in-JS 适合动态主题，Shadow DOM 真隔离。|

## Web安全
### 网站安全防范
| 攻击 | 原理 | 防范 |
|---|---|---|
| **[XSS](/docs/Web开发与安全/XSS攻击)** | 用户输入恶意脚本，页面执行时偷数据 | 转义输出、CSP 头、不用 innerHTML |
| **[CSRF](/docs/Web开发与安全/CSRF攻击)** | 用户在 A 站登录后点击钓鱼链接，冒用身份请求 A 站接口 | Token 验证、SameSite Cookie、Referer 校验 |
| **[中间人](/docs/来时路/计算机网络八股#https如何保证安全)** | 攻击者截获 HTTP 明文请求，篡改数据 | HTTPS 加密传输 |

**总结**：XSS 防脚本注入，CSRF 防冒用身份，HTTPS 防偷听。前端安全面试三板斧。
