## 鉴权设计
### [双token无感刷新](/blog/tech/双token无感刷新的实现.md)如果refresh token也失效了，但这时挂起了一堆请求，刷新失败了，怎么处理?
:::info **思路：请求队列 + 失败重定向**
当 refresh token 也失效时，本质是登录态彻底过期。这时候挂起的请求再刷新也没意义了，所以直接拒绝这些挂起的请求，并跳转到登录页。
:::
#### 1. 请求挂起队列
在发起 refresh 请求时，将这段时间内所有并发请求拦截下来，**不直接拒绝也不放行**，暂存入一个队列：
```ts
let isRefreshing = false
let pendingQueue: Array<{ resolve: Function; reject: Function }> = []

axios.interceptors.response.use(
  response => response,
  async error => {
    const { config, response } = error
    if (response?.status !== 401 || config._retry) return Promise.reject(error)

    // 已有 refresh 在进行 → 挂起
    if (isRefreshing) {
      return new Promise((resolve, reject) => {
        pendingQueue.push({ resolve, reject })
      })
    }

    config._retry = true
    isRefreshing = true

    try {
      const { token } = await axios.post('/auth/refresh', {
        refreshToken: getRefreshToken(),
      })
      setAccessToken(token)

      // 刷新成功 → 重放挂起的请求
      pendingQueue.forEach(({ resolve }) => resolve(axios(config)))
      pendingQueue = []
      return axios(config)
    } catch {
      // refresh 也失败 → 拒绝所有挂起请求
      pendingQueue.forEach(({ reject }) => reject(error))
      pendingQueue = []
      window.location.href = '/login'  // 跳登录页
      return Promise.reject(error)
    } finally {
      isRefreshing = false
    }
  }
)
```

#### 2. 执行流程图
```
请求 A (401)
  ↓
发现 refresh token 存在 → 发起 refresh
  ↓
请求 B (401)          ← A 的 refresh 还没返回
  ↓
isRefreshing = true → 挂起 B
  ↓
请求 C (401)
  ↓
挂起 C
  ↓
refresh 返回
  ├── 成功 → 更新 token → 依次重放 A/B/C
  └── 失败 → 清空队列 → 跳转 /login
```

#### 3. 边界处理
| 场景 | 做法 |
|---|---|
| refresh 失败但不是 401（如网络断开） | 不要跳登录，让用户手动刷新重试 |
| 多 tab 同步 | 用 `localStorage` + `broadcastChannel` 同步刷新状态 |
| 登录/注册接口 401 | 配置 `_skipAuth: true` 白名单，不触发刷新 |

### [JWT登录机制的原理](/docs/Web开发与安全/JWT身份认证.md)
#### JWT结构
`header.payload.signature`，三部分都是 Base64 编码，合起来是一段字符串。

#### 登录流程
```txt
用户登录 → 服务端验证账号密码 → 签发 JWT（含 userId、过期时间） → 返回前端
前端收到 JWT → 存 localStorage → 每次请求带在 Authorization header 里
服务端收到请求 → 验证 JWT 签名 → 解析 payload → 放行或拒绝
```

#### 为什么JWT可以无状态
JWT的签名由服务端密钥生成，只要密钥不泄露，服务端不需要存session，解析JWT就能确认身份。

#### access + refresh双token机制
| | access token | refresh token |
|---|---|---|
| 有效期 | 短（15min~1h） | 长（7天~30天） |
| 存在哪 | 前端内存或 localStorage | localStorage |
| 用途 | 请求鉴权 | 刷新 access token |
| 风险 | 泄露了影响范围小 | 泄露了要立即撤销 |

**为什么不用单 token**：单 token 有效期短则频繁登录体验差，有效期长则泄露风险大。双 token 用短期 access token 保证安全，用长期 refresh token 保证体验。

### 为什么使用JWT? 相比传统session方案有什么优势?
| | Session | JWT |
|---|---|---|
| 存储位置 | 服务端内存/Redis | 客户端持有，服务端不存 |
| 扩展性 | 多台服务器需共享 session（Redis） | 天然支持水平扩展，任何服务器都能验 |
| 请求开销 | 每次查 session（可能查 DB/Redis） | 直接验签名，无需查存储 |
| 跨域 | 难处理（Cookie 跨域受限） | 放 Authorization header，天然跨域 |
| 实时撤销 | 服务端删除 session 即可 | 需维护黑名单或缩短有效期 |

- **JWT 核心优势**：**无状态**。服务端不需要存任何登录信息，验签名就能确认身份，多机部署不用共享 session。
- **JWT 核心劣势**：**不可撤销**。签出去的 token 在到期前永远有效，无法主动踢人，只能等过期或维护黑名单。

**总结**：JWT 无状态，服务器不存 session，直接验签名即可确认身份，天然支持水平扩展。代价是签发后不可撤销，只能等过期。

## 微前端
### 集成一个第三方的摄像监控系统，用iframe嵌进去，而且还要求根据用户权限控制展示内容，怎么处理?
:::info **核心矛盾**
iframe 跨域隔离 → 无法直接操作第三方内容 + 需要按权限动态展示。
:::
#### 方案一：URL token + 第三方配合（推荐）
第三方提供接口，前端通过 URL 参数传递用户权限标识：
```ts
// 前端：根据权限生成带 token 的 URL
const token = generateJwt({ userId, role, cameras: user.cameraIds })
setIframeSrc(`https://监控系统.com/embed?token=${token}`)
```
第三方在 iframe 内自行解析 token，只渲染有权限的摄像头。

#### 方案二：postMessage 动态控制
第三方支持 postMessage 协议，前端按权限动态发指令：
```ts
const iframe = document.querySelector('iframe')

function updatePermissions(cameraIds: string[]) {
  iframe.contentWindow?.postMessage(
    { type: 'update-cameras', cameraIds },
    'https://监控系统.com'
  )
}

window.addEventListener('message', (e) => {
  if (e.origin !== 'https://监控系统.com') return
  console.log('iframe 状态:', e.data)
})
```

#### 方案三：多 iframe + 条件渲染
第三方不配合时的兜底方案：
```tsx
function CameraPanel({ user }: { user: User }) {
  const availableCameras = CAMERAS.filter(c => user.cameraIds.includes(c.id))
  return (
    <div className="grid grid-cols-2 gap-4">
      {availableCameras.map(cam => (
        <iframe key={cam.id} src={`https://监控系统.com/camera/${cam.id}`} />
      ))}
    </div>
  )
}
```

#### 方案四：服务端代理转发
后端做权限校验，代理请求第三方视频流，前端只负责展示。

#### 总结
| 方案 | 可控性 | 依赖第三方 | 推荐场景 |
|---|---|---|---|
| URL token | 高 | 需配合 | **首选** |
| postMessage | 中 | 需支持协议 | 已有 postMessage 接口 |
| 多 iframe | 低 | 不需要 | 兜底 |
| 服务端代理 | 最高 | 需提供 API | 安全性要求高 |


## 数据结构
### 如果你从多个系统的API拉数据，但这些系统的数据结构不一致，比如A系统返回camelCase、B系统是snake_case，还有一些是中文key，怎么统一数据结构来复用?

:::info **核心思路**
防腐层（Anti-Corruption Layer）+ 统一 Schema

**API防腐层**：对外部系统的数据结构做一层适配，转换成内部统一的标准结构。上层业务只认标准结构，不关心外部系统的差异。也就是*在外部系统和业务代码之间加一层隔离，防止外部系统的"脏东西"污染核心代码。*
:::
内部定义一套标准结构（推荐 camelCase），给每个外部系统写一个适配器做转换，上层业务只认标准结构。

#### 1. 定义标准 Schema
```ts
// 内部统一数据结构
interface User {
  userId: string
  userName: string
  createdAt: string
}
```

#### 2. 按数据源写适配器
```ts
// A 系统：camelCase → 直接映射
function adaptFromSystemA(raw: any): User {
  return {
    userId: raw.userId,
    userName: raw.userName,
    createdAt: raw.createdAt,
  }
}

// B 系统：snake_case → 转换键名
function adaptFromSystemB(raw: any): User {
  return {
    userId: raw.user_id,
    userName: raw.user_name,
    createdAt: raw.created_at,
  }
}

// C 系统：中文 key → 映射
function adaptFromSystemC(raw: any): User {
  return {
    userId: raw['用户ID'],
    userName: raw['用户名'],
    createdAt: raw['创建时间'],
  }
}
```

#### 3. 统一调用入口
```ts
const adapters = {
  systemA: adaptFromSystemA,
  systemB: adaptFromSystemB,
  systemC: adaptFromSystemC,
}

async function fetchUsers(source: keyof typeof adapters) {
  const raw = await apiMap[source].getUsers()
  return adapters[source](raw)  // 返回统一结构
}
```

#### 4. 通用工具：递归转 camelCase
```ts
import { camelCase, mapKeys, mapValues } from 'lodash-es'

function toCamelCase(obj: any): any {
  if (Array.isArray(obj)) return obj.map(toCamelCase)
  if (obj !== null && typeof obj === 'object') {
    return mapKeys(
      mapValues(obj, toCamelCase),
      (_, key) => camelCase(key),
    )
  }
  return obj
}
```

#### 总结
```
外部数据 → 适配器(Adapter) → 统一 Schema → 业务层
  ↑              ↑               ↑
蛇形/中文/...   各自写一份       只认一套
```

**一句话**：对内定义标准接口，对外用适配器做转换，不改上游、不改下游、只加一层。

## 组件设计
### 如果项目有20+个页面，需求方经常改字段、加字段，每次都要改组件，怎么设计?
:::info **核心思路**
配置驱动 + 递归渲染组件
:::
把页面描述为 JSON 配置，组件根据配置自动渲染，改字段只需要改配置，不用改组件。

#### 1. 定义配置结构
```ts
// 描述一个表单/详情页的配置
interface FieldConfig {
  key: string
  label: string
  type: 'input' | 'select' | 'date' | 'image' | 'table'
  required?: boolean
  hidden?: boolean
  options?: { label: string; value: string }[]
  children?: FieldConfig[]  // 嵌套子表
  render?: (value: any, record: any) => ReactNode  // 自定义渲染
}
```

#### 2. 递归渲染组件
```tsx
function DynamicField({ field, value, onChange }: Props) {
  if (field.hidden) return null

  switch (field.type) {
    case 'input':
      return <Input label={field.label} value={value} onChange={onChange} />
    case 'select':
      return <Select label={field.label} options={field.options} value={value} onChange={onChange} />
    case 'table':
      return (
        <SubTable config={field.children} value={value} onChange={onChange} />
      )
    case 'image':
      return <ImageUpload label={field.label} value={value} onChange={onChange} />
    default:
      return field.render?.(value, { field })
  }
}
```

#### 3. 页面由配置驱动
```ts
// 需求改字段 → 只改这个配置对象
const userFormConfig: FieldConfig[] = [
  { key: 'name', label: '姓名', type: 'input', required: true },
  { key: 'gender', label: '性别', type: 'select', options: [...] },
  { key: 'avatar', label: '头像', type: 'image' },
  { key: 'education', label: '教育经历', type: 'table', children: [...] },
]

function UserPage() {
  const [form, setForm] = useState({})
  return (
    <Form>
      {userFormConfig.map(field => (
        <DynamicField
          key={field.key}
          field={field}
          value={form[field.key]}
          onChange={v => setForm(prev => ({ ...prev, [field.key]: v }))}
        />
      ))}
    </Form>
  )
}
```

#### 4. 加字段流程
```
旧流程：需求加字段 → 找页面 → 改 JSX → 改类型 → 改接口 → 改验证
新流程：需求加字段 → 在配置数组加一项
```

#### 5. 进阶：服务端下发配置
如果字段改动非常频繁，配置甚至可以不下发代码，由后端接口返回：
```ts
// 接口同时返回数据和页面配置
const { data, fields } = await api.get('/page/user')
// data 是表单数据，fields 是渲染配置
return <PageRenderer fields={fields} data={data} />
```

#### 总结
:::success **面试要点**
不要说"用配置驱动"，要说出具体怎么做的——*JSON Schema 描述页面、递归组件渲染、配置与服务分离、变更只改一处。*
:::

### 如果要做一个拖拽式仪表盘Dashboard，用户能自己拖拽、组合图表组件展示不同数据源的信息，怎么设计?
:::info **核心思路**
栅格布局 + 组件注册机制 + 配置持久化
:::

#### 1. 整体架构
```
组件库 (Widget Registry) ← 注册 → 拖拽面板 (Dashboard Editor)
       ↑                              ↓
  内置图表 / 自定义组件         生成 JSON 配置
                                    ↓
                              配置持久化 (保存到后端)
                                    ↓
                             渲染器 (Dashboard Renderer)
```

#### 2. 组件注册机制
每种图表/组件预先注册到系统中：
```ts
interface WidgetMeta {
  type: string          // 唯一标识，如 'line-chart'
  name: string          // 显示名，如 '折线图'
  component: React.ComponentType<WidgetProps>
  defaultW: number      // 默认宽 (栅格单位)
  defaultH: number      // 默认高
  configurable?: FieldConfig[]  // 可配置属性
}
```

#### 3. 布局数据结构
```ts
interface DashboardConfig {
  id: string
  name: string
  widgets: WidgetInstance[]
}

interface WidgetInstance {
  id: string
  type: string          // 组件类型，从注册表取
  x: number
  y: number
  w: number
  h: number
  props: Record<string, any>
  dataSource?: {
    api: string
    params: Record<string, any>
  }
}
```

#### 4. 关键库选型
| 需求 | 推荐库 | 说明 |
|---|---|---|
| 拖拽 + 栅格布局 | `react-grid-layout` | 支持拖拽、缩放、响应式 |
| 图表渲染 | `echarts` / `recharts` | 按需选一个 |
| 拖拽排序 | `dnd-kit` | 侧边栏组件列表拖拽 |

#### 5. 渲染器实现
```tsx
function DashboardRenderer({ config }: { config: DashboardConfig }) {
  return (
    <GridLayout
      layout={config.widgets.map(w => ({
        i: w.id, x: w.x, y: w.y, w: w.w, h: w.h,
      }))}
      cols={12}
      rowHeight={60}
    >
      {config.widgets.map(widget => {
        const meta = widgetRegistry[widget.type]
        if (!meta) return null
        return (
          <div key={widget.id}>
            <meta.component {...widget.props} dataSource={widget.dataSource} />
          </div>
        )
      })}
    </GridLayout>
  )
}
```

#### 6. 编辑模式 vs 查看模式
```tsx
function Dashboard({ mode }: { mode: 'edit' | 'view' }) {
  if (mode === 'edit') return <DashboardEditor />  // 可拖拽
  return <DashboardRenderer />                       // 只展示
}
```

#### 7. 数据源管理
每张图表绑定不同数据源，统一适配：
```ts
async function fetchWidgetData(source: WidgetInstance['dataSource']) {
  const raw = await fetch(source.api, { params: source.params })
  return normalizeData(raw)  // 统一成 { labels: [], values: [] }
}
```

#### 总结
:::success **面试要点**
不要只说"用 react-grid-layout"，而是讲清楚*组件注册机制、配置与渲染分离、数据源抽象、编辑/查看双模式。*
:::

## 性能优化
### 如果现在一个页面要同时展示300辆车的实时位置+轨迹，怎么优化?
:::info **核心瓶颈**
300 辆车 × 实时更新 × 轨迹点 → DOM 爆炸 + 渲染卡顿
:::

#### 方案一：Canvas 渲染替代 DOM（首选）
DOM 元素超过几百个就会卡，Canvas 是位图绘制，几千个点无压力：
```tsx
// 使用 Canvas 覆盖在地图上绘制车辆和轨迹
function VehicleLayer({ vehicles }: { vehicles: Vehicle[] }) {
  const canvasRef = useRef<HTMLCanvasElement>(null)

  useEffect(() => {
    const ctx = canvasRef.current?.getContext('2d')
    if (!ctx) return

    // 清空画布
    ctx.clearRect(0, 0, ctx.canvas.width, ctx.canvas.height)

    // 批量绘制轨迹 + 车辆图标，避免频繁 DOM 操作
    vehicles.forEach(v => {
      // 画轨迹线
      ctx.beginPath()
      ctx.strokeStyle = v.color
      ctx.lineWidth = 2
      v.trail.forEach((p, i) => i === 0 ? ctx.moveTo(p.x, p.y) : ctx.lineTo(p.x, p.y))
      ctx.stroke()

      // 画车辆图标（简化为圆形）
      ctx.beginPath()
      ctx.arc(v.x, v.y, 6, 0, Math.PI * 2)
      ctx.fillStyle = v.color
      ctx.fill()
    })
  }, [vehicles])

  return <canvas ref={canvasRef} className="absolute inset-0" />
}
```

#### 方案二：数据层面的优化
```ts
// 1. 只传增量数据，不传全量
ws.onmessage = (e) => {
  const diff = JSON.parse(e.data)  // 只包含位置变化的车辆
  updateVehicles(diff)
}

// 2. 降低精度：后端返回坐标保留 4 位小数（≈11m 精度）就够用
//    原始: 113.123456789 → 传输 15 字节
//    优化: 113.1234       → 传输 8 字节，省一半

// 3. 轨迹点采样：每 5 秒一个点，路径平滑足够
//    1 秒一个点 → 300 车 × 60 点/分 × 每分钟刷新 = 18000 点
//    5 秒一个点 → 300 车 × 12 点/分 = 3600 点，减少 80%
```

#### 方案三：视口裁剪 + 聚合
```ts
// 只渲染可视区域内的车辆
function getVisibleVehicles(vehicles: Vehicle[], bounds: BBox) {
  return vehicles.filter(v =>
    v.x >= bounds.minX && v.x <= bounds.maxX &&
    v.y >= bounds.minY && v.y <= bounds.maxY
  )
}

// 缩放级别低时，聚合相邻车辆
function clusterVehicles(vehicles: Vehicle[], zoom: number) {
  if (zoom > 14) return vehicles  // 高缩放级，展示单个
  // 用网格聚合：将相近车辆合并为一个聚合点
  return gridCluster(vehicles, getGridSize(zoom))
}
```

#### 方案四：Web Worker 分担计算
把坐标转换、聚合计算放到 Worker 线程，不阻塞主线程渲染：
```ts
// worker.ts
self.onmessage = ({ data }) => {
  const result = clusterVehicles(data.vehicles, data.zoom)
  self.postMessage(result)
}

// 主线程
const worker = new Worker(new URL('./worker.ts', import.meta.url))
worker.postMessage({ vehicles, zoom })
worker.onmessage = ({ data }) => setClustered(data)
```

#### 方案对比
| 维度 | DOM 方案 | Canvas 方案 | WebGL 方案 |
|---|---|---|---|
| 渲染数量 | &lt; 500 | 5000+ | 10万+ |
| 交互（点击/悬浮） | 天然支持 | 需做命中检测 | 复杂 |
| 实现复杂度 | 低 | 中 | 高 |
| 推荐场景 | 简单图表 | **300 辆车最合适** | 上亿点云 |

#### 总结
:::success **面试要点**
不要只说"用 Canvas"，而是要分层讲：*数据层（增量更新+精度压缩）、渲染层（Canvas替代DOM）、可视层（视口裁剪+聚合）、计算层（Worker 分担）。*
:::

### 重排和重绘是怎么引发的? 怎么解决?
- **重排（Reflow）**：改变元素尺寸/位置/布局属性时，浏览器重新计算布局树。
- **重绘（Repaint）**：改变颜色/背景等不影响布局的属性时，浏览器重新绘制。

#### 引发重排的操作
| 操作 | 示例 |
|---|---|
| 修改几何属性 | `width`、`height`、`padding`、`margin` |
| 修改位置 | `top`、`left`、`display: none`、浮动 |
| 读写布局属性 | `offsetWidth`、`clientHeight`、`getComputedStyle` |
| 增删 DOM | `appendChild`、`removeChild` |
| 窗口缩放 | `resize` 事件 |

#### 重绘不引发重排
`color`、`background-color`、`visibility`、`opacity`、`box-shadow`。

#### 优化方案
| 问题 | 解决 |
|---|---|
| 频繁改样式 | 合并用 `classList` 一次性修改，或 `cssText` 批量赋值 |
| 频繁读布局属性 | 先用变量存起来再操作（避免强制同步布局） |
| 批量 DOM 操作 | 用 `DocumentFragment` 或 `display: none` 批量改完再显示 |
| 动画卡顿 | 用 `transform` + `opacity` 做动画（GPU 合成，不触发重排） |
| 复杂动画 | 让元素脱离文档流（`position: fixed/absolute`），不影响其他元素 |

#### 总结
重排必重绘，重绘不一定重排。优化核心是减少布局属性读写、合并样式操作、用 transform 替代 top/left。|

## 计算机网络
### 页面要同时订阅车辆位置、报警事件、设备状态，怎么复用WS连接且不丢数据?
:::info **核心原则**
一个 WS 连接 + 按消息 type 分发 + 断线缓存 + seq 校验补数据
:::

#### 1. 架构
```
WS连接(单例) → 分发器 → 车辆位置组件(subscribe 'position')
                      → 报警组件(subscribe 'alarm')
                      → 设备状态组件(subscribe 'status')
```

#### 2. WSClient 单例 + 按 type 分发
```ts
type Handler = (data: any) => void

class WSClient {
  private ws: WebSocket | null = null
  private handlers = new Map<string, Set<Handler>>()
  private offlineBuffer: any[] = []   // 断线缓存
  private lastSeq = 0                  // 服务端消息的递增 seq

  connect(token: string) {
    this.ws = new WebSocket(`wss://api.com/ws?token=${token}`)

    this.ws.onmessage = (e) => {
      const msg = JSON.parse(e.data)   // { type, seq, data }
      this.lastSeq = msg.seq
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
    }

    this.ws.onclose = () => {
      setTimeout(() => this.connect(token), 3000)
    }

    this.ws.onopen = () => {
      this.flushBuffer()                // 重连成功 → 消费缓存
      this.fetchMissed()                 // 拉取断线期间遗漏数据
    }
  }

  // 组件调用
  subscribe(type: string, handler: Handler) {
    if (!this.handlers.has(type)) this.handlers.set(type, new Set())
    this.handlers.get(type)!.add(handler)
    this.send({ type: 'subscribe', channels: [type] })
    return () => this.handlers.get(type)?.delete(handler)
  }

  // 重连后：先消费本地缓存，再 HTTP 补拉遗漏
  private async flushBuffer() {
    while (this.offlineBuffer.length) {
      const msg = this.offlineBuffer.shift()!
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
    }
  }
  private async fetchMissed() {
    const res = await fetch(`/api/ws/missed?from=${this.lastSeq + 1}`)
    const list = await res.json()
    list.forEach((msg: any) => {
      this.handlers.get(msg.type)?.forEach(cb => cb(msg.data))
      this.lastSeq = msg.seq
    })
  }
}
```

#### 3. 组件使用
```tsx
function VehicleMap() {
  useEffect(() => wsClient.subscribe('position', setVehicles), [])
}
function AlarmPanel() {
  useEffect(() => wsClient.subscribe('alarm', v => setAlarms(p => [...p, v])), [])
}
function DeviceStatus() {
  useEffect(() => wsClient.subscribe('status', setDevices), [])
}
```
三个组件共享一个 WS 连接，互不干扰。

#### 4. 不丢数据的保证
| 阶段 | 机制 |
|---|---|
| 连接正常 | 消息带递增 `seq`，客户端可检测跳号 |
| 连接断开 | 数据缓存在 `offlineBuffer` |
| 重连成功 | 先消费缓存，再 `HTTP GET /api/ws/missed?from=seq` 补拉遗漏 |
| 组件卸载 | cleanup 取消订阅，不影响其他组件 |

#### 5. 多 Tab 场景：SharedWorker
不需要每个 tab 都建一个 WS：
```ts
const ws = new WebSocket('wss://api.com/ws')
const ports = new Set<MessagePort>()
const cache: any[] = []

ws.onmessage = (e) => {
  const msg = JSON.parse(e.data)
  cache.push(msg)
  ports.forEach(p => p.postMessage(msg))
}

self.onconnect = (e) => {
  const port = e.ports[0]
  ports.add(port)
  cache.forEach(msg => port.postMessage(msg))
  port.onclose = () => ports.delete(port)
}
```

#### 总结
:::success **面试要点**
答这个题的关键是"**知道怎么防丢**"：*单连接 + 按 type 分发解决多连接浪费、seq 递增检测跳号、断线缓存 + HTTP 补拉保证不丢、SharedWorker 解决多 tab 重复建连。*
:::

### HTTP请求方法有哪些? 各自什么场景?
#### [8种方法](/docs/来时路/计算机网络八股/#常见的http请求方法)
| 方法 | 含义 | 典型场景 |
|---|---|---|
| GET | 获取资源 | 查询列表、获取详情 |
| POST | 提交新资源 | 创建订单、提交表单 |
| PUT | 全量更新 | 更新用户全部信息 |
| PATCH | 部分更新 | 只修改用户头像 |
| DELETE | 删除资源 | 删除一条记录 |
| HEAD | 只获取响应头 | 检查资源是否存在、获取文件大小 |
| OPTIONS | 查询支持的请求方法 | 跨域预检请求（CORS preflight） |
| CONNECT | 建立隧道 | HTTPS 代理 |

#### 幂等性
| 方法 | 幂等 | 说明 |
|---|---|---|
| GET / HEAD / OPTIONS | ✅ | 读操作，多次请求结果不变 |
| PUT / DELETE | ✅ | 多次更新/删除同一个资源，结果一致 |
| POST | ❌ | 每次创建新资源，多次调用会创建多条 |
| PATCH | ❌ | 部分更新可能依赖当前状态 |

#### RESTful规范
增 POST → 删 DELETE → 改 PUT/PATCH → 查 GET

### [同源策略](/docs/来时路/浏览器原理八股#同源策略)是什么? 怎么解决[跨域](/docs/计算机网络/跨域)?
**定义**：浏览器的安全机制，限制不同源（协议、域名、端口任一不同）的文档或脚本互相访问。

#### 限制的内容
- ❌ 跨域 DOM 访问（`iframe` 无法操作不同源的页面内容）
- ❌ 跨域 AJAX 请求（`XMLHttpRequest` / `fetch` 默认被拦截）
- ✅ 跨域资源引用不受限（`<script>`、`<img>`、`<link>` 可以加载）

#### 跨域解决方案
| 方案 | 原理 | 适用场景 |
|---|---|---|
| CORS | 服务端返回 `Access-Control-Allow-Origin` 头 | **标准方案，首选** |
| [JSONP](/docs/情景与手撕题/实现jsonp) | 利用 `<script>` 不受同源策略限制 | 仅 GET 请求，兼容老旧浏览器 |
| 反向代理 | 同源路径代理到目标服务器（如 Vite/webpack proxy） | 开发环境，或 Nginx 转发 |
| postMessage | `window.postMessage` 跨文档通信 | iframe 通信、跨域窗口通信 |
| WebSocket | 协议本身支持跨域 | 实时通信 |

#### CORS响应头字段
| 响应头 | 说明 | 示例 |
|---|---|---|
| `Access-Control-Allow-Origin` | 允许的源 | `*` 或 `https://example.com` |
| `Access-Control-Allow-Methods` | 允许的请求方法 | `GET, POST, PUT, DELETE` |
| `Access-Control-Allow-Headers` | 允许的自定义头 | `Authorization, Content-Type` |
| `Access-Control-Allow-Credentials` | 是否允许带 cookie | `true`（不能和 `*` 同时用） |
| `Access-Control-Max-Age` | 预检结果缓存时间（秒） | `86400` |

**简单请求 vs 非简单请求**：简单请求（GET/POST/HEAD + 无自定义头 + Content-Type 限 `text/plain`/`multipart`/`x-www-form-urlencoded`）直接发；非简单请求（PUT/DELETE/自定义头/`application/json`）先发 OPTIONS 预检。

#### [JSONP原理](/docs/情景与手撕题/实现jsonp)
动态创建 `<script>` 标签，请求 URL 带 `?callback=fn`，服务端返回 `fn({data})`，浏览器作为 JS 直接执行拿到数据。仅支持 GET。

**可跨域加载资源的标签**：`<script>`、`<img>`、`<link>`（仅 CSS）、`<video>`/`<audio>`、`<iframe>`。浏览器允许加载但**不允许 JS 读取内容**（除非 CORS）。

#### 代理服务器跨域原理
跨域限制是浏览器的安全策略，服务端之间没有跨域限制。浏览器请求同域代理地址，代理转发到目标服务器再返回结果——浏览器全程只跟自己的域通信，不触发同源策略。

### 点击图表某个项目，地图跟着联动展示，怎么处理?
:::info **核心思路**
统一状态管理 + 事件总线 + 数据过滤
:::

#### 1. 整体架构
```
图表点击 → 更新全局选中状态 → 地图监听状态变化 → 地图高亮/聚焦
                    ↓
              其他面板（详情、列表等）同步更新
```

#### 2. 统一状态管理
不要在各组件内部各自维护状态，而是抽到上层统一管理：
```ts
// 全局选中状态
interface GlobalSelection {
  selectedId: string | null
  selectedType: "city" | "vehicle" | "device" | null
  hoveredId: string | null
  filter: Record<string, any>
}

const selectionStore = {
  state: { selectedId: null, selectedType: null, hoveredId: null, filter: {} },
  listeners: new Set<() => void>(),

  setSelected(id: string, type: string) {
    this.state.selectedId = id
    this.state.selectedType = type
    this.notify()
  },

  setFilter(filter: Record<string, any>) {
    this.state.filter = filter
    this.notify()
  },

  subscribe(cb: () => void) {
    this.listeners.add(cb)
    return () => this.listeners.delete(cb)
  },

  notify() {
    this.listeners.forEach(cb => cb())
  },
}
```

#### 3. 图表组件：点击时发布
```tsx

}
```

## 文件处理
### 大文件分片上传的原理，如何分片，断点续传，如何记录上传进度?
:::info
**核心流程**：文件切片 → 并发上传 → 服务端合并 → 断点续传依赖唯一标识
:::

#### 1. 整体流程
```
文件选择 → 计算 hash → 切分成片 → 并发上传 → 服务端合并
                    ↓                       ↑
              记录已上传的片             断点续传(跳过已上传)
```

#### 2. 文件分片
```ts
const CHUNK_SIZE = 5 * 1024 * 1024  // 每片 5MB

function createChunks(file: File): Blob[] {
  const chunks: Blob[] = []
  let start = 0
  while (start < file.size) {
    const end = Math.min(start + CHUNK_SIZE, file.size)
    chunks.push(file.slice(start, end))
    start = end
  }
  return chunks
}
```

#### 3. 计算文件 hash（唯一标识）
用 `spark-md5` 在浏览器端计算文件 hash，用于断点续传时服务端识别同一文件：
```ts
import SparkMD5 from 'spark-md5'

function calcFileHash(file: File): Promise<string> {
  return new Promise((resolve, reject) => {
    const spark = new SparkMD5.ArrayBuffer()
    const reader = new FileReader()
    const chunks = createChunks(file)

    let i = 0
    reader.onload = (e) => {
      spark.append(e.target!.result as ArrayBuffer)
      i++
      if (i < chunks.length) {
        reader.readAsArrayBuffer(chunks[i])
      } else {
        resolve(spark.end())  // 得到文件唯一 hash
      }
    }
    reader.onerror = reject
    reader.readAsArrayBuffer(chunks[0])
  })
}
```

#### 4. 上传 + 进度记录
```ts
async function uploadFile(file: File) {
  const fileHash = await calcFileHash(file)
  const chunks = createChunks(file)
  const total = chunks.length
  let uploaded = 0

  // 先问服务端：哪些片已经上传过（断点续传）
  const { uploadedIndexes } = await fetch(`/api/upload/status?hash=${fileHash}`).then(r => r.json())
  const uploadedSet = new Set(uploadedIndexes)

  // 并发上传未上传的分片（限制并发数）
  const concurrency = 3
  const tasks = chunks.map((chunk, index) => async () => {
    if (uploadedSet.has(index)) {
      uploaded++  // 跳过已上传的
      updateProgress(uploaded / total)
      return
    }
    const formData = new FormData()
    formData.append('chunk', chunk)
    formData.append('hash', fileHash)
    formData.append('index', String(index))
    formData.append('total', String(total))

    await fetch('/api/upload/chunk', { method: 'POST', body: formData })
    uploaded++
    updateProgress(uploaded / total)
  })

  // 并发控制
  await concurrentLimit(tasks, concurrency)

  // 通知服务端合并
  await fetch('/api/upload/merge', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ hash: fileHash, total }),
  })
}

function updateProgress(percent: number) {
  // 更新进度条
  progressBar.style.width = `${(percent * 100).toFixed(1)}%`
}
```

#### 5. 断点续传的关键
```
上传中断 → 重新选择同一文件（hash 相同）
         → 请求 /api/upload/status?hash=xxx
         → 服务端返回已收到的分片索引 [0, 1, 3, 5, ...]
         → 客户端只上传缺失的分片 [2, 4, 6, ...]
         → 合并时校验完整性
```

#### 6. 服务端伪代码
```ts
// /api/upload/status — 查询哪些片已上传
app.get('/api/upload/status', (req, res) => {
  const { hash } = req.query
  const uploaded = redis.smembers(`upload:${hash}`)  // 存 Set 结构
  res.json({ uploadedIndexes: uploaded.map(Number) })
})

// /api/upload/chunk — 接收单片
app.post('/api/upload/chunk', upload.single('chunk'), (req, res) => {
  const { hash, index } = req.body
  const chunkPath = path.join(UPLOAD_DIR, hash, index)
  fs.mkdirSync(path.dirname(chunkPath), { recursive: true })
  fs.renameSync(req.file.path, chunkPath)
  redis.sadd(`upload:${hash}`, index)
  res.json({ ok: true })
})

// /api/upload/merge — 合并所有分片
app.post('/api/upload/merge', (req, res) => {
  const { hash, total } = req.body
  const writeStream = fs.createWriteStream(path.join(UPLOAD_DIR, `${hash}.zip`))
  for (let i = 0; i < total; i++) {
    writeStream.write(fs.readFileSync(path.join(UPLOAD_DIR, hash, i)))
  }
  writeStream.end()
  res.json({ url: `/files/${hash}.zip` })
})
```

#### 总结
:::success **面试要点**
分片上传的核心是*前端 hash 去重 + 分片并发 + 服务端按索引合并*。断点续传依赖 hash 标识已上传的片，跳过重传。进度 = 已上传片数 / 总片数。
:::

## React
### [useEffect和useLayoutEffect的区别](/docs/来时路/React八股#useeffect与uselayouteffect的区别)
:::info **核心区别**
执行时机不同
:::

| | `useEffect` | `useLayoutEffect` |
|---|---|---|
| 执行时机 | 浏览器绘制**之后**异步执行 | 浏览器绘制**之前**同步执行 |
| 阻塞绘制 | 不阻塞 | 会阻塞 |
| 适用场景 | 数据请求、日志、订阅 | DOM 测量、动画、需同步读布局 |

**执行顺序**：组件渲染 → DOM 更新 → `useLayoutEffect`（绘制前） → 浏览器绘制 → `useEffect`（绘制后）

**怎么选**：不涉及 DOM 测量就用 `useEffect`（默认）；需要在绘制前拿到 DOM 尺寸或执行动画就用 `useLayoutEffect`，否则用户会看到闪烁。

### [Vue和React的diff算法区别](/docs/来时路/React八股#react与vue的diff算法区别)
**相同点**：都是基于虚拟DOM的同层比较，不做跨层级复用。

**关键区别**：
| | Vue | React |
|---|---|---|
| 对比策略 | 双端指针（头尾往中间比） | 从左到右单次遍历 |
| key 的作用 | 判断可复用的节点（移动而非重建） | 判断可复用的 Fiber 节点 |
| 列表更新 | 头尾各两个指针，尽可能复用 | 只从头开始，标记移动位置 |
| 更新粒度 | 组件级 + 响应式追踪（精确到值） | Fiber 架构 + 时间分片（整棵树 diff） |

**本质差异**：
- **Vue**有响应式系统，知道哪个数据变了、变了多少，diff时直接锁定变化范围，双端指针尽可能复用旧节点，减少移动
- **React**没有响应式追踪，不知道数据哪变了，只能整棵树重diff，靠Fiber分片执行、可中断

**一句话**：Vue的diff是"知道哪变了再去比"（响应式驱动）；React的diff是"不知道哪变了所以全比一遍"（无侵入式）。

### Vue和React的区别
| 维度 | Vue | React |
|---|---|---|
| 范式 | 声明式 + 响应式 | 函数式 + 不可变 |
| 数据流 | 双向绑定（v-model） | 单向数据流 |
| 状态管理 | data 直接修改（响应式代理） | setState(不可变更新) |
| 模板 | 模板语法 + 指令（v-if/v-for） | JSX（JavaScript in HTML） |
| diff 策略 | 双端指针 + 静态标记 | Fiber + 单端遍历 |
| 更新粒度 | 响应式追踪，精确到组件 | 顶往下全量 diff，Fiber 可中断 |
| 上手难度 | 低（模板直观） | 中（JSX + 函数式概念） |
| 适用场景 | 后台管理、快速交付 | 大型复杂应用、需精细控制 |

**核心差异**：
- **响应式**：Vue 用 Proxy 代理数据，修改 data 自动触发更新；React 需要手动 setState，用不可变数据触发重新渲染
- **模板 vs JSX**：Vue 模板接近 HTML，有 v-if/v-for 指令；React 全 JS，`if/for` 就是 JS 本身的语法
- **更新机制**：Vue 知道哪个数据变了，精确更新对应组件；React 不知道哪变了，从根组件开始整棵树 diff，靠 Fiber 分片不卡 UI

## JavaScript
### [数组浅拷贝](/docs/JavaScript/freeCodeCamp笔记#数组的浅拷贝)的方法有哪些?
4种方法：`[...arr]`、`Array.from(arr)`、`arr.slice()`、`[].concat(arr)`。都只拷贝第一层，嵌套对象引用不变。

[深拷贝](/docs/情景与手撕题/实现深拷贝)用 `JSON.parse(JSON.stringify())`（有坑）或 `structuredClone()`（原生）。

### [闭包](/docs/来时路/JavaScript八股#对闭包的理解)的使用场景
**定义**：函数 + 它引用的外部变量，即使外部函数执行完，内部函数还能访问那些变量。

#### 常用场景
| 场景 | 原理 | 示例 |
|---|---|---|
| 数据私有化 | 变量不暴露到全局 | 计数器、缓存 |
| [函数柯里化](/docs/情景与手撕题/柯里化currying的实现) | 固定部分参数，返回新函数 | `const add = x => y => x + y` |
| [防抖/节流](/docs/Web开发与安全/防抖与节流) | 保存 timer 引用 | 搜索输入、滚动事件 |
| [高阶组件（React）](/docs/来时路/React八股#高阶组件hoc) | 记住外部传入的配置 | HOC 模式 |
| 循环中的异步 | 用 IIFE 保存每次循环的 i | `for` 中 `setTimeout` |

#### 闭包陷阱
循环中用闭包取到的变量是最终值而非当前值，需要用IIFE（立即调用函数表达式，immediately invoked function expression）或`let`解决。

#### 总结
**一句话**：闭包 = 函数 + 它"记住"的外部变量，用于封装私有数据、函数工厂、防抖节流。

## Webpack
### webpack打包优化怎么做? loader如何选型?
#### 打包优化方向
| 方向 | 手段 |
|---|---|
| 体积 | Tree Shaking、代码分割（SplitChunks）、压缩（TerserPlugin）、按需加载 |
| 速度 | 缓存（cache-loader /持久化缓存）、多进程（thread-loader / esbuild）、缩小范围（include/exclude） |
| 图片/静态资源 | 小图转 base64（url-loader）、压缩（image-webpack-loader） |

#### Loader 选型原则
| 处理类型 | Loader | 说明 |
|---|---|---|
| JS/TS | `babel-loader` / `esbuild-loader` | esbuild 更快，babel 生态更成熟 |
| CSS | `css-loader` + `style-loader` | 开发环境用 style-loader，生产用 MiniCssExtractPlugin |
| 预处理器 | `sass-loader` / `less-loader` | 配合 css-loader 链式使用 |
| 图片字体 | `url-loader` / `asset-module` | webpack5 推荐用内置 Asset Module，小图转 base64 |
| Vue | `vue-loader` | 解析 .vue 单文件 |
| SVG | `@svgr/webpack` | 可当 React 组件使用 |

**Loader执行顺序**：从右到左，从下到上。例如 `sass-loader → css-loader → style-loader`。

#### 面试重点优化项：
1. **代码分割**：`SplitChunksPlugin` 提取公共依赖，`import()` 按路由懒加载
2. **多进程构建**：`thread-loader` 把耗时 loader 放到子进程
3. **持久化缓存**：webpack5 内置 `cache: { type: 'filesystem' }`，二次构建秒出
4. **Tree Shaking**：`sideEffects: false` + ESM 模块，去掉未引用代码

## 浏览器原理
### [浏览器缓存机制](/docs/来时路/浏览器原理八股#浏览器缓存机制)是什么? [强缓存和协商缓存](/docs/来时路/浏览器原理八股#强缓存与协商缓存)有哪些?
#### 强缓存
**强缓存**：不请求服务端，直接读本地缓存。
| 响应头 | 说明 |
|---|---|
| `Expires` | 绝对过期时间（HTTP/1.0，依赖客户端时间） |
| `Cache-Control: max-age=3600` | 相对过期秒数（HTTP/1.1，**优先级高**） |

强缓存命中 → 状态码 `200 (from disk/memory cache)`

#### 协商缓存
**协商缓存**：带缓存标识问服务端，服务端判断是否可用。
| 请求头 | 响应头 | 说明 |
|---|---|---|
| `If-Modified-Since` | `Last-Modified` | 文件最后修改时间 |
| `If-None-Match` | `ETag` | 文件唯一指纹（**优先级高**） |

协商缓存命中 → `304 Not Modified`；未命中 → `200` 返回新资源。

#### 总结
- **缓存流程**：请求 → 有强缓存? → 是：直接用本地缓存。否：发请求带 `If-None-Match` → 服务端判断没变 → 304；变了 → 200。
- **刷新行为**：地址栏回车/链接跳转 → 强+协商都生效；F5 → 仅协商生效；Ctrl+F5 → 全不生效。
